# Dev Implementation Plan: Commit, Merge, and Branch Model

**Date:** 11 Mar 2026
**Version:** v0.13.20
**Depends on:** Architect review v0.13.20

---

## Overview

Transform the vault from a "sync-and-overwrite" model to a proper commit/push/pull/merge workflow that supports multi-user collaboration. Also restructure the CLI into `vault` and `vault dev` subgroups.

```python
        # 2. Read FETCH_HEAD tree (theirs)

        # 4. Compute 3-way diff
        # 5. Apply clean merges
        # 6. Handle conflicts with .conflict.ext rename
        # 7. Create merge commit with parents = [local_HEAD, FETCH_HEAD]
        # 8. Update HEAD and refs/remotes/origin/main
        # 9. Return {added, modified, deleted, conflicts}

    def three_way_diff(self, base_tree, local_tree, remote_tree) -> dict:
        # Returns: {
        #   clean_local:  [paths only changed locally],
        #   clean_remote: [paths only changed remotely],
        #   conflicts:    [paths changed in both],
        #   local_adds:   [paths added only locally],
        #   remote_adds:  [paths added only remotely],
        #   local_deletes: [paths deleted only locally],
        #   remote_deletes: [paths deleted only remotely],
        #   conflict_deletes: [paths where one side edited, other deleted]
        # }

    def resolve_conflict(self, directory, path, local_blob_id, remote_blob_id,
                          object_store, read_key) -> dict:
        # 1. Determine conflict filename: path.conflict.ext
        # 2. Rename local file to conflict filename
        # 3. Write remote version to original path
        # 4. Return {original_path, conflict_path}
```

**Conflict filename logic:**
```python
def conflict_path(self, original_path: str) -> str:
    # "report.pdf"         → "report.conflict.pdf"
    # "src/main.py"        → "src/main.conflict.py"
    # "README"             → "README.conflict"
    # "data.tar.gz"        → "data.tar.conflict.gz"
    parts = original_path.rsplit('.', 1)
    if len(parts) == 2:
        return f'{parts[0]}.conflict.{parts[1]}'
    return f'{original_path}.conflict'
```

### B4: Tracking Merge Base

**Important detail:** To do a 3-way merge, we need to know the "base" — the last point where local and remote agreed.

**Strategy:** Before fetch updates `refs/remotes/origin/main`, save its current value as the merge base. This is the tree state at the last sync point.

```
Before fetch:
  refs/heads/main         = local_commit_5
  refs/remotes/origin/main = sync_commit_3  ← this is the merge base

After fetch:
  FETCH_HEAD              = remote_commit_7
  refs/remotes/origin/main = remote_commit_7
  MERGE_BASE              = sync_commit_3   ← saved before overwrite
```

Store as `.sg_vault/MERGE_BASE` (temporary, cleared after merge).

## Phase C: Push Guards

### C1: Safe Push

```python
def push(self, directory: str) -> dict:
    # 1. Check working directory is clean
    #    → "Uncommitted changes. Run 'vault commit' first."
    # 2. Fetch latest remote state
    # 3. Compare refs/remotes/origin/main with local history
    #    - If remote HEAD is ancestor of local HEAD → safe to push
    #    - If not → "Remote has diverged. Run 'vault pull' first."
    # 4. Upload only new/changed blobs (diff local tree vs remote tree)
    # 5. Build and upload tree.json
    # 6. Upload settings.json
    # 7. Create push commit (or reuse existing HEAD commit)
    # 8. Update refs/remotes/origin/main
    # Return: {pushed_commit, blobs_uploaded, summary}
```

### C2: Ancestry Check

```python
def is_ancestor(self, object_store, read_key, ancestor_id, descendant_id) -> bool:
    # Walk commit chain from descendant back to root
    # Return True if ancestor_id is found in the chain
    current = descendant_id
    while current:
        if current == ancestor_id:
            return True
        commit = self.read_commit(object_store, read_key, current)
        current = commit.parents[0] if commit.parents else None
    return False
```

## Open Questions for Human Review

1. **Should `vault commit` also encrypt+store blobs, or defer blob upload to push?**
   - If commit stores blobs locally: commit is self-contained, push just uploads
   - If commit only records tree: commit is fast, but push must do more work
   - **Dev recommendation:** Commit stores blobs locally. Push uploads delta.

2. **Auto-commit on push (`--auto-commit` flag)?**
   - Convenience for single-user workflows
   - Equivalent to `vault commit -m "auto" && vault push`
   - **Dev recommendation:** Yes, but not the default behavior.

3. **How to handle the merge base when there's no prior sync?**
   - First pull after clone: merge base = clone commit
   - Edge case: vault was created with `init`, never pulled
   - **Dev recommendation:** If no refs/remotes/origin/main exists, treat the root commit as merge base.

4. **Should we support `vault diff` to show what would be committed?**
   - Natural complement to explicit commit
   - **Dev recommendation:** Yes, add in Phase A. Reuses existing status logic.

5. **Conflict file naming: what about files with multiple dots?**
   - `data.tar.gz` → `data.tar.conflict.gz` or `data.conflict.tar.gz`?
   - **Dev recommendation:** Insert `.conflict` before the last extension: `data.tar.conflict.gz`

6. **Version number in tree.json — increment on commit or push?**
   - Currently: incremented on push (in `_build_tree_json`)
   - With explicit commit: version should increment on push (server-facing version)
   - Local commits don't change the server version
   - **Dev recommendation:** Version only increments on push. Commits track lineage via parent pointers.
