## What Was Done This Session

1. **`content_hash` computation** — `Vault__Crypto.content_hash()` computes SHA256(plaintext)[:12]. `Vault__Sync.commit()` and `status()` now use content_hash instead of file size for change detection. This fixes the bug where same-size edits were invisible.

2. **Tree entry field encryption** — `Vault__Commit` now encrypts `name`/`size`/`content_hash` into `name_enc`/`size_enc`/`content_hash_enc` (base64-encoded AES-GCM) when serializing trees, and decrypts them on load. This establishes the foundation for future multi-level key derivation where different keys could encrypt different directory levels.

3. **New safe type** — `Safe_Str__Encrypted_Value` for base64-encoded encrypted field values.

4. **New tests** — 4 new tests for content_hash detection, encrypted field verification, and blob reuse.

## Key Architectural Decisions Made

| Decision | Rationale |
|----------|-----------|
| SHA256(plaintext)[:12] for content_hash | Reliable change detection without storing full file content; 12 hex chars = 48 bits, sufficient for collision avoidance in typical vaults |
| Field-level encryption in tree entries | Foundation for per-directory key derivation; defense-in-depth over blob-level encryption |
| Clone branch private key stays in `local/` | Never uploaded to vault; proves device identity cryptographically |
| Named branch private key encrypted in vault | Shared among authorized users who have the vault key |
| EC P-256 key pairs per branch | Matches Web Crypto API; ECDSA signatures for commit provenance |
| Opaque IDs everywhere (`obj-`, `ref-`, `idx-`) | Server sees no structure; zero-knowledge by design |
| Uint millisecond timestamps | Deterministic, no timezone ambiguity, matches JavaScript Date.now() |
