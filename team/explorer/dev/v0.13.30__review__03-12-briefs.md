# Developer Review: v0.13.30 Architecture Briefs

**date** 12 Mar 2026
**from** Developer
**re** Three briefs from project lead (bare vault unification, branch model, CLI remotes)

---

## Implementation Perspective

These briefs are ambitious but grounded. The codebase has good bones — Type_Safe enforcement, clean crypto separation, content-addressed objects already working. Here's what I see from the trenches.

---

## 1. Implementation Complexity

### Brief 1 (Vault Unification) — Medium

**Straightforward parts:**
- Defining the unified schema is schema work. We already have Schema__Object_Tree, Schema__Object_Commit, Schema__Object_Tree_Entry. Extending these is mechanical.
- Adding format_version field — trivial.
- Nested vaults (.vault.zip) — just a blob convention. The object store already handles arbitrary binary blobs.

**Hard parts:**
- **Refactoring Vault__Sync** — this is the most coupled class. It bridges tree.json (legacy) and object store (new). ~460 lines with interleaved logic for both formats. Rewriting this without breaking clone/push/pull is the real work.
- **Security audit** — requires mapping every field in both server and local structures. Tedious but necessary. The path encryption decision cascades everywhere.
- **Zip-based clone** — needs server support (caching zip per vault version). CLI side is simple; server side depends on the Transfer API team.

### Brief 2 (Branch Model) — Hard

**Straightforward parts:**
- Adding fields to Schema__Object_Commit (branch_fingerprint, signature) — Type_Safe schema change.
- Branch key generation — PKI__Crypto already handles EC P-256 key generation. Reuse that.
- Signing commits — PKI__Crypto.sign() exists. Wire it into the commit creation path.
- `log --graph` — parsing commit chain and rendering ASCII. Bounded complexity.

**Hard parts:**
- **Merge algorithm** — no existing merge infrastructure. Need to build: tree diff, LCA computation, three-way merge, conflict detection and reporting. This is the core complexity.
- **Multi-parent commits** — Schema__Object_Commit currently has `parent_id: str` (single parent). Merge commits need multiple parents. Schema change + all code that walks the commit chain.
- **Branch storage on server** — how does the server store branches? Current API is `read/write/delete(vault_id, file_id)`. Branches need refs. Server API may need extension.
- **Testing multi-user flows** — simulating two concurrent clones, parallel edits, merge. Integration test complexity is high.

### Brief 3 (CLI Remotes) — Medium-Low

**Straightforward parts:**
- Backend abstraction interface — 4 methods (read/write/delete/list). Clean interface.
- Backend__LocalFolder — filesystem read/write. Simplest possible backend.
- Backend__Zip — zip/unzip wrapper. Python's zipfile module handles it.
- `remote add/remove/list` commands — CLI boilerplate + config storage.
- Vault export as zip — iterate object store, zip it up.

**Hard parts:**
- **Backend__SG_Send adapter** — current Vault__API uses vault_id + file_id addressing, not blob_id. Need mapping layer. Also needs write_key for mutations (HMAC authentication). This is the impedance mismatch.
- **Remote-aware push/pull** — current push/pull assumes SG/Send. Needs to resolve which remote, load config, instantiate correct backend.

## 2. Current Codebase Gaps

### Missing Abstractions

| Gap | Needed For | Effort |
|-----|-----------|--------|
| Backend interface | Remotes (Brief 3) | 1-2 days |
| Tree diff algorithm | Merge (Brief 2) | 2-3 days |
| LCA computation | Merge (Brief 2) | 1-2 days |
| Multi-parent commit schema | Merge (Brief 2) | 1 day |
| Branch refs storage | Branches (Brief 2) | 1-2 days |
| Remote config storage | Remotes (Brief 3) | 1 day |

### Code That Needs Refactoring

1. **Vault__Sync** (~460 lines) — most complex class. Bridges two formats. Must be rewritten for unified format.
2. **Vault__API** — monolithic, hardcoded to SG/Send. Needs to become one backend implementation behind an interface.
3. **CLI__Vault** — currently calls Vault__Sync directly. Needs indirection through remote config for push/pull.

### Type_Safe Schema Gaps

- `Schema__Object_Commit` needs: `parent_ids: list[str]` (multi-parent), `branch_fingerprint: Safe_Str__Key_Fingerprint`, `signature: Safe_Str__Signature`
- New schema needed: `Schema__Branch_Ref` (branch_id, head_commit, public_key, created_at)
- New schema needed: `Schema__Remote_Config` (name, url/path, backend_type, vault_id)
- New Safe_* types needed: `Safe_Str__Branch_Id`, `Safe_Str__Remote_Name`, `Safe_Str__Backend_Type`

## 3. Testing Challenges

### Crypto Interop (Brief 2 — Signed Commits)

CLAUDE.md says: "All crypto operations must produce output that matches the browser (Web Crypto API) byte-for-byte." Branch signing uses ECDSA P-256. Need test vectors that prove Python `cryptography` library produces identical signatures to Web Crypto API given same inputs.

**Challenge:** ECDSA signatures are non-deterministic (random k value). To get byte-identical output, both sides must use deterministic ECDSA (RFC 6979) or the test vectors must use fixed nonces. This needs investigation.

### Multi-User Integration Tests (Brief 2)

Testing the full flow: User A clones, User B clones, both edit, both push, merge. This requires:
- Two separate vault directories
- Two separate branch key pairs
- Simulating concurrent commits
- Verifying merge result

Can't use mocks (per CLAUDE.md rules). Must test against real objects. Can test locally without server by using LocalFolder backend (Brief 3) — which suggests **Brief 3 should ship before Brief 2's tests**.

### Merge Edge Cases

- Both add same file with same content (should auto-merge)
- Both add same file with different content (should conflict)
- One deletes file, other modifies it (should conflict)
- Both delete same file (should auto-merge)
- Rename detection (Brief doesn't mention this — defer?)
- Empty vaults, single-file vaults, deeply nested paths

## 5. Concerns from the Trenches

### Scope Creep Risk

Brief 2 mentions: Mermaid diagrams, `<sg-vault-graph>` web component, multi-vault UI, drag-and-drop between vaults, document-level branching. These are vision items, not v0.13.30 deliverables. **Recommend explicitly scoping them out** to avoid scope creep.

### Type_Safe Ceremony

Every new concept needs a Safe_* type, a Schema class, and a test. Branch model introduces ~5 new Safe_* types, ~3 new schemas. This is correct per CLAUDE.md rules but adds ceremony. Budget time for it.

### Vault__Sync is the Bottleneck

This class is the hardest to refactor. It's where both formats meet. It has the most integration surface. Brief 1 (unification) is essentially "rewrite Vault__Sync." Don't underestimate this.

### Server API Dependencies

- Zip-based clone needs server to generate and cache zips
- Branch refs need server storage (or we store branches purely client-side)
- Push/pull with remotes may need server API changes

**Recommendation:** Design so that branches and refs are stored as regular vault objects (encrypted blobs in the object store). This avoids server API changes and works with any backend.

### ECDSA Determinism

The crypto interop requirement may conflict with ECDSA signature non-determinism. Need to investigate RFC 6979 (deterministic ECDSA) support in both Python `cryptography` and Web Crypto API.

## Recommended Execution Order

```
Week 1:     Quick wins (format_version, remote config, export zip, security audit doc)
Week 2-3:   Vault__Sync refactor (unified format)
Week 3-4:   Backend abstraction + remotes (parallel)
Week 4-5:   Branch key generation + commit signing
Week 6-8:   Merge algorithm + conflict handling
Week 8-9:   Integration testing + polish
```

Start with quick wins to build momentum. Tackle Vault__Sync early (it's the bottleneck). Run remotes in parallel. Save merge for last (hardest, benefits from everything before it).

---

## Summary

| Brief | Verdict | Biggest Risk |
|-------|---------|-------------|
| 1 (Unification) | Do it first. Unblocks everything. | Vault__Sync refactor complexity |
| 2 (Branches) | Achievable but merge is hard. | Merge algorithm + testing |
| 3 (Remotes) | Easiest of the three. Ship alongside Brief 1. | SG/Send API impedance mismatch |

The ideas are sound. The codebase can support them. The main risk is scope — keep v0.13.30 focused on the core mechanics (unified format, backends, branches, basic merge) and save the vision items (web components, document-level branching, drag-and-drop) for later.
