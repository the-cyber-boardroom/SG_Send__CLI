## What's Already Built (Solid Foundation)

- **Vault lifecycle (complete):** init, clone, push, pull, status, remote-status
- **Inspection/debugging tools (complete):** inspect, inspect-object, inspect-tree, inspect-log, cat-object, inspect-stats, derive-keys
- **Crypto layer (complete):** AES-256-GCM, PBKDF2, HKDF, HMAC — Web Crypto interop by design
- **Type system (partially adopted):** 16 Safe_* types, 8 schema classes
- **Object store:** content-addressable storage with commit/tree structure (mirrors git)

## What's Genuinely New

| Area | Status | Effort |

## Architectural Risks

### Risk 1: CLI Module Violates CLAUDE.md Rules
The `cli/__init__.py` has 14+ module-level functions, zero classes. Violates "Classes for everything" rule. Must refactor before expanding.

### Risk 2: Vault__API Uses Raw Primitives
`base_url: str` and `access_token: str` — raw primitives in a Type_Safe class. `Safe_Str__Access_Token` exists but is unused.

### Risk 3: Upload/Download Requires Different API Surface
Current `Vault__API` only knows vault operations. File transfer is a different bounded context. Needs a separate `Transfer__API` class.

### Risk 4: PKI Introduces New Cryptographic Surface
Current crypto is exclusively symmetric. PKI needs asymmetric crypto (key pairs, signing, ECIES encryption). Must be a separate `PKI__Crypto` class. Algorithm choices (Ed25519 vs ECDSA P-256, key formats) are a one-way door.

### Risk 5: Secrets Management Has Platform Dependencies
`keyring` library or self-contained encrypted file store? Encrypted-file approach (using existing `Vault__Crypto`) avoids new dependencies.

### Risk 6: CLI Restructuring Is a Breaking Change
Flat commands → subcommand groups. Need backward-compatible aliases during transition.

## What Needs Upfront Design vs Incremental

### Needs Upfront Design
1. **CLI architecture** — command dispatch pattern, backward compatibility strategy, base command class
2. **PKI algorithm choices** — curves, signature scheme, encryption scheme, key serialization format
3. **Transfer API contract** — endpoint patterns, auth model, multipart upload support

### Can Be Built Incrementally
1. Secrets management (self-contained, no dependency on other features)
2. `vault log` and `vault status` aliases (thin wiring layer)
3. Upload single file (once Transfer API contract known)
4. PKI key generation (isolated from sign/verify/encrypt)

## Architecture Strengths

- **Object store model is extensible** — content-addressable with commit/tree structure
- **Crypto layer is clean** — pure computation, no I/O, no side effects
- **Safe types are reusable** — established pattern for new types

## Architecture Gaps

- **No API base class** — `Vault__API` uses raw urllib. `Transfer__API` and `PKI__API` would duplicate HTTP machinery. Extract `API__Base`.
- **No credential resolution pipeline** — token handling scattered across CLI helper functions. Need `Credential__Resolver` chain: CLI flag → env var → keychain → file.
- **CLI needs complete restructuring** — 14 module-level functions → class-based command handlers.

## Recommended Implementation Sequence

```
Phase 0 (CLI refactor) ─┬──> Phase 2 (vault status/log)  ← quick win
                        ├──> Phase 4 (secrets)            ← self-contained
                        ├──> Phase 1 (upload/download)    ← needs API contract
                        └──> Phase 3 (PKI)                ← needs algorithm decisions
```
