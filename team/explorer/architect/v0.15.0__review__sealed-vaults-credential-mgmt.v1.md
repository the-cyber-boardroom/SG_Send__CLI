# Architecture Review — Sealed Vaults & Credential Management

**Date:** 11 March 2026
**Status:** Design proposal
**Requested by:** Dinis Cruz
**Depends on:** `v0.14.0__review__cloneless-vault-access.md`
**Priority:** High — UX blocker for all vault workflows

---

## Problem Statement

Every vault operation requires the user to type or paste a vault key:

```bash
sg-send-cli clone {24-char-passphrase}:{vault-id}
sg-send-cli ls {24-char-passphrase}:{vault-id}
sg-send-cli cat {24-char-passphrase}:{vault-id} path/to/file
```

This is painful for:
1. **Repeated use** — the same vault key typed dozens of times per session
2. **Automation** — Lambda/CI needs vault keys in env vars or config files (plaintext!)
3. **Multi-vault workflows** — managing 5+ vault keys manually
4. **Security** — keys in shell history, env vars, plaintext config files

### What We Need

A local credential store that:
- Stores vault keys encrypted at rest (never plaintext on disk)
- Unlocks with a single passphrase for the session
- Allows commands to look up vault keys by name/alias
- Works for both interactive CLI and programmatic (Lambda) use

---

## Three Layered Solutions

### Layer 1: `~/.sg-send/` — Global Config Home

A top-level config directory for sg-send-cli, following the XDG/dotfile convention:

```
~/.sg-send/
├── config.json          # default base-url, default token, preferences
├── tokens/              # per-vault access tokens (optional)
└── vaults.enc           # encrypted vault key store (Secrets__Store)
```

**Key insight:** We already have `Secrets__Store` which does exactly this — encrypted key-value store protected by a passphrase. We just need to wire it into the CLI.

### Layer 2: Sealed Vault Mode

**Naming decision:** "Sealed" over "headless" because:
- "Headless" = "no UI" (headless browser, headless Chrome) — misleading
- "Sealed" = "encrypted container, needs key to access" (Hashicorp Vault uses this)
- Natural verb pair: `seal` / `unseal`
- Clear mental model: a sealed envelope vs an open folder

**What a sealed vault is:**
A vault where all data is cloned locally as encrypted objects, but **no plaintext files are ever written to disk**. The `.sg_vault/` object store has everything, but you can only read individual files by decrypting them on-demand.

```
my-vault/                    # regular clone (unsealed)
├── .sg_vault/
│   ├── objects/...          # encrypted blobs
│   ├── refs/HEAD
│   ├── VAULT-KEY            # ← plaintext vault key on disk
│   └── tree.json
├── config.json              # ← plaintext file on disk
└── deploy/EC2.json          # ← plaintext file on disk

my-vault/                    # sealed clone
├── .sg_vault/
│   ├── objects/...          # encrypted blobs (same as above)
│   ├── refs/HEAD
│   └── tree.json            # ← encrypted tree (no VAULT-KEY file!)
└── (no plaintext files)     # nothing else on disk
```

**The key difference:** A sealed vault does NOT store the vault key on disk. To read any file, you must provide the vault key (or have it in your `~/.sg-send/vaults.enc` credential store).

### Layer 3: Zip Container (Portable Sealed Vault)

A sealed vault packed into a single `.sg-vault.zip` file:

```bash
# Clone as a zip
sg-send-cli clone --sealed --zip my-vault.sg-vault.zip {key}:{id}

# Read a file from the zip
sg-send-cli cat --from my-vault.sg-vault.zip path/to/EC2.json
```

**Why this works:**
- The object store is already content-addressable blobs — perfect zip entries
- `objects/ab/ab1234567890` → zip entry `objects/ab/ab1234567890`
- osbot-utils has zip helpers (mentioned by Dinis)
- A single `.sg-vault.zip` file is easy to store in S3, attach to CI, embed in Lambda layers

**Without the vault key, a .sg-vault.zip is just a bag of encrypted blobs** — no filenames, no content, no structure is readable.

## Architecture: Credential Store

### Using Secrets__Store (already built!)

`Secrets__Store` already provides:
- `store(passphrase, key, value)` — encrypt and save
- `get(passphrase, key)` — decrypt and retrieve
- `list_keys(passphrase)` — list stored keys
- `delete(passphrase, key)` — remove

We wire this into a new `CLI__Credential_Store`:

```python
class CLI__Credential_Store(Type_Safe):
    secrets : Secrets__Store    # store_path = ~/.sg-send/vaults.enc

    def add_vault(self, passphrase, alias, vault_key):
        """Store a vault key under a human-friendly alias."""

    def get_vault_key(self, passphrase, alias) -> str:
        """Retrieve a vault key by alias."""

    def list_vaults(self, passphrase) -> list:
        """List all stored vault aliases."""

    def remove_vault(self, passphrase, alias):
        """Remove a stored vault key."""

    def resolve_vault_key(self, args) -> str:
        """Smart resolution: check args.vault_key, then alias lookup, then .sg_vault/VAULT-KEY."""
```

### CLI Session Flow

```bash
# First time: add a vault to the credential store
$ sg-send-cli vault add my-deploy-vault
  Vault key: {paste key}
  Passphrase: ****
  ✅ Saved 'my-deploy-vault' to ~/.sg-send/vaults.enc

# Now use it by alias
$ sg-send-cli ls my-deploy-vault
  Passphrase: ****
  📦 deploy-configs (version 3)
  📂 5 files, 12.3 KB
    ...

# Or unlock once for the session (future: sg-send-agent)
$ export SG_SEND_PASSPHRASE=****
$ sg-send-cli ls my-deploy-vault     # no prompt needed
```

### Vault Management Commands

```bash
sg-send-cli vault add <alias>          # Store a vault key
sg-send-cli vault list                 # List stored vault aliases
sg-send-cli vault remove <alias>       # Remove a stored vault key
sg-send-cli vault show <alias>         # Display vault key (requires passphrase)
```

## Architecture: Sealed Clone

### Clone Modes

```bash
# Regular clone (current behavior) — plaintext files on disk
sg-send-cli clone {key}:{id}

# Sealed clone — encrypted objects only, no plaintext
sg-send-cli clone --sealed {key}:{id}

# Sealed clone as zip
sg-send-cli clone --sealed --zip {key}:{id}
```

### Reading from Sealed Vaults

```bash
# List files in a sealed vault (needs vault key or alias)
sg-send-cli ls ./my-sealed-vault --vault-key {key}
sg-send-cli ls ./my-sealed-vault --alias my-deploy-vault

# Read a file from sealed vault
sg-send-cli cat ./my-sealed-vault path/to/EC2.json --vault-key {key}

# Extract a file from sealed vault to disk
sg-send-cli get ./my-sealed-vault path/to/EC2.json ./local.json --vault-key {key}
```

### Seal/Unseal Operations

```bash
# Seal an existing vault (remove plaintext files, remove VAULT-KEY)
sg-send-cli seal ./my-vault

# Unseal a sealed vault (extract all plaintext files)
sg-send-cli unseal ./my-vault --vault-key {key}
```

### Sealed Vault Internal Structure

A sealed vault differs from a regular vault in two ways:
1. **No VAULT-KEY file** — the vault key is not stored on disk
2. **No plaintext files** — only `.sg_vault/` directory exists
3. **Sealed marker** — `.sg_vault/SEALED` file exists (empty, signals sealed state)

## Security Model

### What's Protected

| Scenario | Vault key on disk? | Plaintext files on disk? | Security level |
|----------|-------------------|--------------------------|----------------|
| Regular clone | Yes (VAULT-KEY) | Yes | Low — disk access = full access |
| Sealed clone | No | No | High — need vault key to read anything |
| Sealed + credential store | In ~/.sg-send/vaults.enc (encrypted) | No | High — need passphrase |
| Sealed + zip | No (zip has no key) | No (zip has encrypted blobs) | High — portable encrypted container |

### Threat Model

- **Disk stolen / leaked** — Sealed vault: attacker gets encrypted blobs, useless without vault key
- **Shell history** — Credential store eliminates typing vault keys in commands
- **Env var leaks** — Passphrase in env var is lower-risk than vault key in env var (passphrase only unlocks the local store, not the remote vault)

## Open Questions

1. **Passphrase caching** — Should we support `SG_SEND_PASSPHRASE` env var for automation? Or is that just moving the problem?
2. **Session agent** — Future: `sg-send-agent` daemon (like ssh-agent) that holds unlocked keys in memory for the session duration?
3. **Zip format** — Standard zip with entries, or a custom archive? Zip has the advantage of `zipfile` stdlib support.
4. **Tree encryption in sealed mode** — Should tree.json also be encrypted in the sealed vault? Currently it's plaintext (reveals file paths/sizes but not content). For true sealed mode, it should be encrypted.
5. **Multiple passphrases** — Should different vaults be able to have different passphrases? Or one master passphrase for all?

---

## Implementation Priority

1. **`~/.sg-send/` + credential store** — Immediate UX win, builds on Secrets__Store
2. **`vault add/list/remove/show` commands** — CLI for credential management
3. **Sealed clone mode** — `clone --sealed`, `seal`, `unseal`
4. **Zip container** — `clone --sealed --zip`
5. **Session agent** — Future (like ssh-agent)
