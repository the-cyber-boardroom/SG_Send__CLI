## 5 Design Decisions Needed (Phase 0)

### Decision 1: File Path Encryption

Paths are currently plaintext in tree.json. **Recommendation: encrypt fully.**

- Brief 3 emphasizes vault portability (any remote). Plaintext paths leak metadata to cloud backends.
- Cost is minimal (tree already encrypted; just extends to path level).
- This is a security invariant that can't be changed without migration.

### Decision 2: Branch Key Storage

For CLI context (no GUI, may run in containers/Lambda):

**Recommendation: filesystem encryption** — branch key encrypted with vault read_key, stored in `.sg_vault/branches/{branch_id}/private.pem`. Design a KeyStore interface as extensibility point for future backends (OS keychain, browser storage).

### Decision 3: Merge Conflict Resolution UI

Encrypted blobs can't be diffed like text. **Recommendation: blob-level conflict detection** with options:

```
Conflict in documents/report.pdf:
  Base:    (blob) abc123
  Yours:   (blob) def456
  Theirs:  (blob) ghi789

Choose: --keep-ours / --keep-theirs / --abort
```

Content-aware merge (paragraph-level) deferred to Phase 4.

### Decision 4: Merge Base Computation (LCA)

Standard BFS/DFS over commit parent pointers. ~50-100 LOC, well-tested. No innovation needed.

### Decision 5: Format Version Field

Add `format_version: int` to vault metadata NOW. Brief 1 says "not a concern" but having it costs nothing and prevents painful migrations later.

## Phase 1: Bare Vault Unification

### Current State

Two vault representations exist:
1. **Server-side:** tree.json + settings.json, paths plaintext, sent as blobs via Vault__API
2. **Local bare:** .sg_vault/objects/ with content-addressed encrypted blobs, Schema__Object_Tree

Bridge logic in `Vault__Sync._read_local_tree_from_objects()` translates between them.

### What Needs to Happen

1. **Security audit** (1 week) — decide on path encryption, map every field
2. **Unified format spec** (3 days) — one schema for both server and local
3. **Migrate Vault__Sync** (5 days) — rewrite sync logic against unified format
4. **Verify end-to-end** (3 days) — all operations identical on server and local

### Why It's Highest Leverage

- Clone becomes `curl vault.zip` (single cached download)
- One security audit surface (not two)
- Push becomes delta-only upload
- Nested vaults fall out naturally (just zip-shaped blobs)
- Unblocks Phases 2 and 3

## Phase 2: CLI Remotes

### Current State

Vault__API is hardcoded to SG/Send. Uses urllib with `{base_url}/api/vault/read|write|delete/{vault_id}/{file_id}`. No remote configuration, no abstraction layer.

### What Needs to Happen

1. **Backend abstraction** — `read(blob_id)`, `write(blob_id, data)`, `delete(blob_id)`, `list_blobs()`
2. **Backend implementations** — SG/Send adapter, LocalFolder, Zip, URL (read-only)
3. **Remote configuration** — `.sg_vault/remotes.json`, `sg-send-cli remote add/remove/list`
4. **Vault export as zip** — iterate object store, create distributable zip

### Impedance Mismatch

SG/Send uses vault_id + file_id; generic backends want blob_id only. Solution: Backend__SG_Send adapter maintains internal mapping. Straightforward, no blocker.

### Most Operations Already Work Offline

Create vault, add files, commit, encrypt/decrypt, sign/verify — all local. Network only needed for push/pull/clone-from-remote. Just need to verify and document this.

## Phase 3: Branch Model & Merge

### Current State

What exists: Schema__Object_Commit (parent, tree_id, timestamp, message), content-addressed object store, PKI infrastructure (PKI__Crypto with key generation, signing, verification).

What's missing: branch identity (no fingerprint field in commits), signatures on commits, merge logic, conflict detection, diff algorithm, log --graph.

### What Needs to Happen

1. **Extend Schema__Object_Commit** (2 days) — add branch_fingerprint, signature fields
2. **Branch key generation on clone** (3 days) — auto-create EC P-256 key pair
3. **Merge algorithm** (7-10 days) — three-way merge at blob level, LCA computation, conflict detection
4. **Log visualization** (3 days) — ASCII graph, Mermaid generation

### The Hard Part: Merge Algorithm

Three-way merge for encrypted vaults:
- Load base tree (LCA), ours tree, theirs tree
- For each unique path across all trees, compare blob_ids
- Auto-merge non-conflicting changes
- Flag conflicts where both sides modified same path differently
- Build merged tree from results

~200-300 LOC, but needs heavy testing. Allocate best developer.

### Signature Trust Model

- Signing: mandatory on create (can't create unsigned commit)
- Verification: optional on pull (client can verify, or trust remote)
- Merge verification: informative (show who signed what)
- Key rotation: new key for future commits; past signatures are immutable historical records

## Architectural Risks

1. **Merge algorithm complexity** — biggest risk. Blob-level merge is achievable; content-level merge is Phase 4+
2. **File path encryption decision** — affects everything downstream. Must decide in Phase 0
3. **Type_Safe constraints** — Safe_* types add ceremony but enforce correctness. New schema fields (branch_fingerprint, signature) need Safe_* types
4. **Web Crypto interop** — brief says all crypto must match browser byte-for-byte. Branch signing (ECDSA P-256) must use same parameters as Web Crypto API

---

## What Excites Me Most

1. **Unified format = force multiplier** — eliminates format duality, simplifies everything
2. **Branch model makes collaboration possible** — conflicts at merge boundaries only, not at commit time
3. **Mandatory signed commits** — stronger than Git's optional GPG, built into the format
4. **Offline-first is rare** — most collaborative systems need constant sync; this doesn't
5. **Remotes = vault portability** — vaults travel via email, zip, S3, not trapped in one platform
6. **The fractal vision** — vault-level branches today, document-level tomorrow, concept-level eventually

---

## Codebase Health

**Strengths:** Type_Safe throughout, clean crypto separation, content-addressed object store, good test coverage, modular CLI structure, no Pydantic/mocks.

**Weaknesses:** Format duality (tree.json vs object tree), no merge/diff, hardcoded SG/Send API, linear-only commit chain, size-only file modification detection.

**Bottom line:** Solid foundation. Well-organized. Ready for these three briefs.
