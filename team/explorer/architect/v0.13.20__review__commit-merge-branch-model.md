# Architect Review: Commit, Merge, and Branch Model

**Date:** 11 Mar 2026
**Version:** v0.13.20
**Triggered by:** Multi-user vault editing producing data loss; commit history shows only "push"/"pull" with no meaningful context

---

## 1. Problem Statement

The current vault sync model has fundamental issues for multi-user collaboration:

1. **No conflict detection.** Push overwrites the remote tree unconditionally. If User A and User B both clone, edit different files, and push — the second push silently drops the first user's changes.
2. **No explicit commit.** Every push/pull auto-creates a commit. Users can't batch logical changes or describe what they did. Log output is just `push push pull push pull clone`.
3. **No separation of fetch vs merge.** Pull is atomic: download + overwrite local. There's no way to inspect what changed remotely before accepting it.
4. **Modification detection is weak.** Only file size is compared (line 253 of `Vault__Sync.py`). Same-size edits go undetected.

These aren't edge cases — multi-user simultaneous editing is the primary workflow.

---

## 2. Current Architecture Analysis

### What We Already Have (Git-Like Foundations)

The codebase already has strong git-like primitives:

| Concept | Implementation | Git Equivalent |
|---------|---------------|----------------|
| Content-addressable objects | `Vault__Object_Store` (SHA256-based IDs) | `.git/objects/` |
| Commit objects | `Schema__Object_Commit` (parent, tree_id, message, timestamp) | Commit objects |
| Tree objects | `Schema__Object_Tree` (list of path + blob_id entries) | Tree objects |
| Blob objects | Encrypted file content in object store | Blob objects |
| HEAD reference | `Vault__Ref_Manager` (refs/head file) | `.git/refs/heads/main` |
| Commit chain | Parent pointers forming linked list | Commit DAG (linear only) |

### What's Missing

| Concept | Current State | Needed |
|---------|--------------|--------|
| Explicit commit | Auto-created on push/pull | User-triggered with message |
| Remote tracking ref | None — version number in tree.json | `refs/remotes/origin/main` |
| Fetch | Doesn't exist | Download remote state without merging |
| Merge | Doesn't exist | Combine local + remote trees |
| Conflict detection | None | Detect same-file edits |
| Conflict resolution | None | `.conflict.{ext}` rename strategy |
| Branches | Only HEAD | Named refs (refs/heads/{name}) |
| Content hashing for change detection | Size-only comparison | Hash-based comparison |

## 3. Proposed Model

### 3.1 Explicit Commit

**Current flow:** edit files → `push` (auto-creates commit + uploads)
**Proposed flow:** edit files → `commit` (creates local commit) → `push` (uploads to server)

```
Working Directory  →  commit  →  Local HEAD  →  push  →  Remote
                                     ↑
Remote  →  fetch  →  FETCH_HEAD  →  merge  →  Local HEAD
```

**Commit creates:**
1. New tree object from working directory state
2. New commit object with user message, parent = current HEAD
3. Updates HEAD to point to new commit

**No staging area** — commit always snapshots the entire working directory (minus `.sg_vault/` and dotfiles). This matches the user's explicit requirement.

### 3.2 Ref System Expansion

Current: only `refs/head`

Proposed layout:
```
.sg_vault/
├── refs/
│   ├── heads/
│   │   └── main           ← local HEAD (commit ID)
│   ├── remotes/
│   │   └── origin/
│   │       └── main       ← last known remote HEAD (commit ID)
│   └── FETCH_HEAD         ← latest fetched remote HEAD (commit ID)
└── HEAD                   ← symbolic ref: "ref: refs/heads/main"
```

**Migration:** Current `refs/head` maps to `refs/heads/main`. On first use, auto-migrate.

### 3.3 Fetch

Downloads the remote tree and creates a commit object representing the remote state, stored as `FETCH_HEAD` (and updates `refs/remotes/origin/main`). Does NOT touch the working directory or local HEAD.

```python
def fetch(self, directory):
    # 1. Download tree.json + settings.json from server
    # 2. Build Schema__Object_Tree from remote state
    # 3. Download any new blob objects we don't have locally
    # 4. Create commit object for remote state
    # 5. Store as FETCH_HEAD and refs/remotes/origin/main
    # Return: {remote_commit_id, changes_summary}
```

### 3.4 Merge

Combines local HEAD tree with FETCH_HEAD tree using a 3-way merge strategy:

```
        base (common ancestor)
       /                      \
  local HEAD              FETCH_HEAD
       \                      /
        merged result
```

**Finding common ancestor:** Walk both commit chains backward until finding a shared commit ID. For the initial version, the common ancestor is the commit at which the local branch last synced with remote (the previous `refs/remotes/origin/main`).

**Merge outcomes:**

| Local Change | Remote Change | Result |
|-------------|---------------|--------|
| No change | No change | Keep as-is |
| No change | Modified | Accept remote (fast-forward) |
| Modified | No change | Keep local |
| Added | Not present | Keep local add |
| Not present | Added | Accept remote add |
| Deleted | Not present | Keep local delete |
| Not present | Deleted | Accept remote delete |
| Modified | Modified | **CONFLICT** |
| Deleted | Modified | **CONFLICT** |
| Modified | Deleted | **CONFLICT** |
| Added (same path) | Added (same path) | **CONFLICT** (unless identical content) |

**Conflict resolution strategy** (per user spec):

1. Detect conflict
2. Rename local file: `filename.ext` → `filename.conflict.ext`
3. Accept remote version as `filename.ext`
4. Report conflict to user
5. Merge succeeds (no blocked state)
6. User manually resolves by inspecting `.conflict.ext` files, then commits

```python
def merge(self, directory):
    # 1. Read local HEAD tree
    # 2. Read FETCH_HEAD tree
    # 3. Find common ancestor tree (refs/remotes/origin/main before fetch)
    # 4. 3-way diff: base vs local, base vs remote
    # 5. For clean merges: apply changes
    # 6. For conflicts: rename local → .conflict.ext, write remote version
    # 7. Create merge commit with TWO parents: [local_HEAD, FETCH_HEAD]
    # 8. Update HEAD
    # Return: {added, modified, deleted, conflicts: [{path, local_path, remote_path}]}
```

### 3.5 Pull = Fetch + Merge

```python
def pull(self, directory):
    fetch_result = self.fetch(directory)
    if not fetch_result['has_changes']:
        return {'up_to_date': True}
    return self.merge(directory)
```

### 3.6 Push Guards

Push must enforce:

1. **Clean working directory required.** No uncommitted changes (prevents the "push silently includes random edits" bug).
2. **Up-to-date with remote.** Local HEAD must be a descendant of the remote HEAD (i.e., `refs/remotes/origin/main` must be an ancestor of local HEAD). If not → "rejected: pull first".
3. **Version check.** Compare remote version with expected version to detect concurrent pushes.

```python
def push(self, directory):
    # 1. Check status — must be clean (no uncommitted changes)
    # 2. Fetch latest remote state
    # 3. Check if local HEAD descends from remote HEAD
    #    - If not: "Error: remote has diverged. Run 'vault pull' first."
    # 4. Upload changed blobs
    # 5. Upload new tree.json
    # 6. Update refs/remotes/origin/main to local HEAD
```

### 3.7 Change Detection Fix

Replace size-only comparison with content hash:

```python
# Current (broken for same-size edits):
if len(content) != old_file_map[path].get('size', -1):
    modified.add(path)

# Proposed:
import hashlib
content_hash = hashlib.sha256(content).hexdigest()[:12]
if content_hash != old_file_map[path].get('content_hash', ''):
    modified.add(path)
```

This requires adding `content_hash` to tree entries (alongside `blob_id` which is hash of *encrypted* content, not useful for detecting local changes).

## 4. Branches

### Why It's Close

The architecture already supports branches because:
- Commits form a DAG via parent pointers
- HEAD is just a file pointing to a commit ID
- Object store is content-addressable (shared across branches)

### What's Needed for Branches

1. **Named refs:** `refs/heads/{branch_name}` — each stores a commit ID
2. **Symbolic HEAD:** `HEAD` file contains `ref: refs/heads/main` (not a commit ID directly)
3. **Branch operations:**
   - `branch create <name>` — creates `refs/heads/<name>` pointing to current HEAD
   - `branch list` — enumerate `refs/heads/*`
   - `branch switch <name>` — update HEAD symbolic ref + restore working directory from that branch's tree
   - `branch delete <name>` — remove ref
4. **Remote branch tracking:** `refs/remotes/origin/{branch_name}`
5. **Server-side branch support:** The server currently stores one tree.json per vault. For branches, we need per-branch trees on the server.

### Server-Side Constraint (Key Decision)

The current API is: `PUT /api/vault/write/{vault_id}/{file_id}`

Branches need one of:
- **Option A: Encode branch in file_id.** E.g., `tree_file_id` becomes `tree_file_id__branch_{name}`. No server changes. Works with current API.
- **Option B: Server-side branch support.** New endpoints. Requires server changes.

**Recommendation:** Option A for initial implementation. Branches are just different tree.json files stored under branch-encoded file_ids. This means the server is oblivious to branches — all the logic is client-side.

### Branch Merge

Once merge works for fetch/pull, branch merge is the same algorithm:
- Find common ancestor of two branch HEADs
- 3-way merge trees
- Create merge commit with two parents

## 5. Commit Schema Changes

Current:
```python
class Schema__Object_Commit(Type_Safe):
    parent    : Safe_Str__Object_Id
    tree_id   : Safe_Str__Object_Id
    timestamp : Safe_Str__ISO_Timestamp
    message   : Safe_Str__Commit_Message
    version   : Safe_UInt__Vault_Version
```

Proposed:
```python
class Schema__Object_Commit(Type_Safe):
    parents   : list[Safe_Str__Object_Id]    # Was: single parent. Now: list (supports merge commits)
    tree_id   : Safe_Str__Object_Id
    timestamp : Safe_Str__ISO_Timestamp
    message   : Safe_Str__Commit_Message
    version   : Safe_UInt__Vault_Version
    author    : Safe_Str__Author = None      # Optional: user identity when available
```

**Key change:** `parent` (single) → `parents` (list). A normal commit has 1 parent. A merge commit has 2. Root commit has 0.

**Backward compatibility:** Read old commits by treating single `parent` field as `parents: [parent]` (or `[]` if empty).

## 7. Implementation Phases

### Phase A: Foundation (Ref System + Commit)
1. Expand `Vault__Ref_Manager` to support named refs and symbolic HEAD
2. Migrate `refs/head` → `refs/heads/main`
3. Add explicit `commit` command to `Vault__Sync`
4. Add content hashing to change detection
5. Update commit schema (parent → parents, backward-compatible reader)

### Phase B: Fetch + Merge
1. Implement `fetch()` — downloads remote state, stores as FETCH_HEAD
2. Implement `merge()` — 3-way tree merge with conflict detection
3. Implement conflict resolution (`.conflict.ext` rename)
4. Rewrite `pull()` as `fetch() + merge()`

### Phase C: Push Guards
1. Add "clean working directory" check before push
2. Add "up-to-date with remote" check before push
3. Push only uploads diff (blobs that remote doesn't have)

### Phase D: Branches (Future)
1. Symbolic HEAD support
2. Branch create/list/switch/delete
3. Branch-encoded server file_ids
4. Branch merge (reuses Phase B merge)

### Phase E: CLI Restructuring
1. `vault` group: init, clone, commit, push, pull, fetch, merge, status, log
2. `vault dev` subgroup: inspect, inspect-object, inspect-tree, inspect-log, inspect-stats, cat-object, derive-keys, remote-status
3. `pki` group: unchanged
4. `secrets` group: future

## 8. Key Decisions Required

### Decision 1: Commit message — required or optional?
- **Option A:** Required (like `git commit -m "..."`)
- **Option B:** Optional with auto-generated default (e.g., "Updated 3 files")
- **Recommendation:** Optional with auto-generated default. Lower friction for simple workflows, but allows meaningful messages for important changes.

### Decision 2: How to find common ancestor for merge?
- **Option A:** Walk both commit chains backward looking for shared commit ID (correct but potentially slow for long histories)
- **Option B:** Use `refs/remotes/origin/main` before fetch as the merge base (fast, works for the common case, but breaks if history is manipulated)
- **Recommendation:** Option B for now. It's simple, correct for the normal workflow, and avoids walking potentially long commit chains.

### Decision 3: What about uncommitted changes during pull?
- **Option A:** Reject pull if working directory is dirty ("commit or discard first")
- **Option B:** Stash-like behavior (auto-commit dirty state, merge, then re-apply)
- **Recommendation:** Option A. Simpler, explicit, and avoids surprising behavior. User does: `vault commit -m "wip"` → `vault pull` → continue working.

### Decision 4: Merge commit message format?
- **Option A:** Auto-generated: `"Merge remote changes (3 files updated, 1 conflict)"`
- **Option B:** User-provided
- **Recommendation:** Auto-generated. Merge is a mechanical operation.

### Decision 5: Branch name encoding for server storage?
- **Option A:** `{tree_file_id}__br_{branch_name}` — simple but pollutes the file_id namespace
- **Option B:** Use a branch manifest file on the server that maps branch names to tree_file_ids
- **Recommendation:** Option A initially, Option B when we need proper branch management.

### Decision 6: Should push auto-commit?
- **Option A:** Push requires clean state (explicit commit first)
- **Option B:** Push auto-commits if there are changes (current behavior, but with a proper commit message prompt)
- **Recommendation:** Option A for correctness. But offer `vault push --auto-commit` as convenience shortcut.

### Decision 7: How to handle the legacy tree.json server format?
The server currently uses a nested tree.json format that's different from our flat Schema__Object_Tree. Two options:
- **Option A:** Keep tree.json as the canonical server format, build object trees from it locally
- **Option B:** Store our Schema__Object_Tree format on the server, keep legacy tree.json as compatibility layer only
- **Recommendation:** Keep Option A for now. The server format is shared with the web app. Changing it requires coordinated changes.

### Decision 8: Per-file change detection — content hash or encrypt-and-compare?
- **Option A:** Hash plaintext with SHA256, store in tree entry — fast comparison
- **Option B:** Re-encrypt and compare blob_id — uses existing infrastructure but re-encryption produces different ciphertexts (random IV)
- **Recommendation:** Option A. Store `content_hash = SHA256(plaintext)[:12]` in tree entries.

## 9. Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| Backward incompatible commit schema (parent → parents) | Existing vaults can't read new commits | Write migration code that handles both formats |
| Server doesn't support concurrent writes | Two simultaneous pushes could corrupt tree.json | Version check before push; reject if remote version > expected |
| Object store grows without bounds | No garbage collection | Future work; objects are small (encrypted blobs) |
| Merge algorithm edge cases | Subtle data loss | Comprehensive test suite with multi-user scenarios |
| Legacy tree.json / Schema__Object_Tree drift | Inconsistent state | Always regenerate tree.json from Schema__Object_Tree on push |

- Merge is deterministic: same inputs always produce same tree

## 11. Summary

The vault already has 80% of the git-like infrastructure. The key missing pieces are:

1. **Explicit commit** (separates "snapshot local state" from "upload to server")
2. **Remote tracking ref** (knows what the server looked like last time we synced)
3. **3-way merge** (detects and resolves conflicts)
4. **Push guards** (prevents the "overwrite other people's work" bug)

Branches are a natural extension once merge works — it's the same algorithm applied to two named refs instead of local+remote.

The conflict resolution strategy (`.conflict.ext` rename) is pragmatic: it never blocks the merge, preserves both versions, and lets the user decide. This is better than git's inline conflict markers for binary files and non-technical users.

**Estimated complexity:** Phases A+B+C are ~1,500 lines of production code + ~1,000 lines of tests. Phase D (branches) adds ~500 more. This is a significant but well-scoped change.
