## Key Insight: All Vaults Are Already Encrypted At Rest

The `.sg_vault/` directory is always a bag of encrypted blobs. By design, **every clone is already "sealed"** — the objects are AES-256-GCM encrypted and useless without the vault key.

What we call a "regular clone" is really two things:
1. **The vault** — `.sg_vault/` with encrypted objects, refs, tree.json
2. **A working copy** — plaintext files extracted/projected onto the filesystem

Plus two convenience files:
- `.sg_vault/VAULT-KEY` — the vault key stored in plaintext (convenience)
- `.sg_vault/token` — the access token stored in plaintext (convenience)

**The "UI" of a vault is the filesystem.** A "headless" vault has no UI — no files extracted, no working copy. Just the encrypted object store.

This maps directly to git:

| git | sg-send-cli |
|-----|-------------|
| `git clone` | `sg-send-cli clone` (vault + working copy) |
| `git clone --bare` | `sg-send-cli clone --bare` (vault only, no working copy) |
| Bare repos on GitHub/GitLab | Bare vaults on S3 / in zip files |
| `.git/` directory | `.sg_vault/` directory |
| Working tree | Extracted plaintext files |
| `git checkout` | `sg-send-cli checkout` (extract files from vault) |

### Naming Decision

The v1 of this document proposed "sealed/unseal" terminology. Dinis correctly pointed out that:
- "Sealed" implies the vault is locked — but it's **always** encrypted
- The real distinction is: does the vault have a working copy (extracted files) or not?
- This is exactly `git clone` vs `git clone --bare`
- "Headless" is valid — git servers host headless/bare repos, and what we store on S3 is a headless vault
- The filesystem IS the "head" (the UI) — so headless = no filesystem projection

We adopt **"bare"** as the primary term (matches git) with **"headless"** as an acceptable synonym.

## Vault Forms: Three Ways a Vault Can Exist

A vault can live anywhere on the filesystem in one of three forms:

| Form | What's on disk | Use case |
|------|---------------|----------|
| **Working copy** | `.sg_vault/` + plaintext files + VAULT-KEY | Local editing (current default) |
| **Bare (headless)** | `.sg_vault/` only, no plaintext, no VAULT-KEY | Servers, CI, storage |
| **Zip** | Single `.sg-vault.zip` file | Lambda layers, S3 artifacts, portable exchange |

All three are the **same vault** — same encrypted objects, same commit history. The difference is packaging and whether the "UI" (plaintext files) is projected.

## Problem Statement

Two separate problems:

### Problem 1: Vault Key Management
Every vault operation requires typing/pasting a vault key. Painful for repeated use, multi-vault workflows, and automation. Vault keys end up in shell history, env vars, and plaintext config files.

### Problem 2: No Bare Mode
You can't get just the encrypted vault without also extracting all files. For Lambda/CI/deploy, you only need the vault — the working copy is wasted work and disk space.

---

## Solution: `~/.sg-send/` as the Vault Keyring

The `~/.sg-send/` directory solves Problem 1. It is a **credential store** — not a vault itself, but a locked keyring for vault keys.

```
~/.sg-send/
├── config.json          # default base-url, preferences
└── vaults.enc           # encrypted vault key store (Secrets__Store)
```

### How It Works

`vaults.enc` uses the existing `Secrets__Store` — a single encrypted file holding all vault keys, locked with a master passphrase.

```bash
# Store a vault key under an alias
$ sg-send-cli vault add my-deploy-vault
  Vault key: {paste key}
  Passphrase: ****
  ✅ Saved 'my-deploy-vault'

# List stored vaults
$ sg-send-cli vault list
  Passphrase: ****
  my-deploy-vault
  staging-configs
  team-secrets

# Use alias anywhere a vault_key is accepted
$ sg-send-cli ls my-deploy-vault         # instead of typing full key
$ sg-send-cli clone my-deploy-vault      # clone by alias
$ sg-send-cli cat my-deploy-vault path/to/EC2.json

# Show the actual vault key
$ sg-send-cli vault show my-deploy-vault
  Passphrase: ****
  abc123def456...:s09yqbpj
```

### Smart Resolution

Any command that takes a vault_key also accepts an alias:
- Contains `:` → treat as vault key directly
- No `:` → look up in `~/.sg-send/vaults.enc`

### Passphrase Resolution Order

1. `SG_SEND_PASSPHRASE` env var (for automation/CI)
2. Interactive prompt via `getpass` (for humans)
3. Future: `sg-send-agent` daemon (like ssh-agent, holds unlocked keys in memory)

### Credential Store Is NOT a Vault

Important distinction: `~/.sg-send/vaults.enc` is a **Secrets__Store** (encrypted key-value file), not an sg-send vault. It doesn't have objects, commits, or trees — it's just a locked dictionary of `{alias: vault_key}` pairs. This keeps it simple and avoids circular dependencies.

## Solution: Bare Clone Mode

### Clone Modes

```bash
# Regular clone: vault + working copy + VAULT-KEY + token
sg-send-cli clone {key}:{id}

# Bare clone: vault only — no working copy, no VAULT-KEY
sg-send-cli clone --bare {key}:{id}
```

### Working Copy Management

```bash
# Extract working copy from bare vault (like git checkout)
sg-send-cli checkout ./my-bare-vault --vault-key {key}

# Remove working copy, keeping bare vault (reverse of checkout)
sg-send-cli clean ./my-vault
```

### Reading from Bare Vaults

```bash
# List files (decrypts tree in memory)
sg-send-cli ls ./my-bare-vault --vault-key {key}

# Read a file to stdout (decrypts one blob)
sg-send-cli cat ./my-bare-vault path/to/EC2.json --vault-key {key}

# Save a file to disk
sg-send-cli get ./my-bare-vault path/to/EC2.json ./local.json --vault-key {key}
```

All of these work with aliases too:
```bash
sg-send-cli ls ./my-bare-vault --alias my-deploy-vault
```

## Solution: Zip Container (Portable Bare Vault)

### Creating Zip Vaults

```bash
# Clone directly into a zip
sg-send-cli clone --bare --zip deploy.sg-vault.zip {key}:{id}

# Pack an existing vault (any form) into a zip
sg-send-cli pack ./my-vault deploy.sg-vault.zip
```

### Reading from Zip Vaults

```bash
sg-send-cli ls --from deploy.sg-vault.zip --vault-key {key}
sg-send-cli cat --from deploy.sg-vault.zip path/to/EC2.json --vault-key {key}
```

### Zip Structure

```
deploy.sg-vault.zip
├── objects/ab/ab1234567890   (encrypted blob)
├── objects/cd/cd9876543210   (encrypted blob)
├── refs/HEAD                 (commit ref)
└── tree.json                 (file manifest — paths & sizes, not content)
```

**A .sg-vault.zip IS a bare sg-send vault** — same as how a `.git` bare repo is just the repo without a working tree.

## Lambda Deployment: Zip as Vault Layer

```python
# Load vault zip from S3 (or Lambda layer at /opt/) — entirely in memory
s3_response = s3.get_object(Bucket='configs', Key='deploy.sg-vault.zip')
zip_bytes   = s3_response['Body'].read()

# Read one config file — no disk I/O, no extraction
vault_zip  = Vault__Zip(crypto=Vault__Crypto())
config_raw = vault_zip.from_bytes(zip_bytes, vault_key, 'configs/EC2.json')
config     = json.loads(config_raw)
```

| Current pattern | Problem |
|----------------|---------|
| Config in env vars | Plaintext, size limits, visible in console |
| N files in S3 | N API calls, plaintext at rest |
| N secrets in Secrets Manager | $0.40/secret/month, N API calls |
| sg-send-cli clone in Lambda | 13s, full disk extraction, /tmp usage |

| Zip vault pattern | Advantage |
|-------------------|-----------|
| 1 zip in S3 + 1 secret (vault key) | Cheap, fast, encrypted |
| In-memory only | No disk I/O, no /tmp cleanup |
| ~0.1s overhead | vs 13s for full clone |
| Lambda layer support | Upload once, shared across functions, versioned |
| Already how Lambda deps work | Teams already load zips from S3 |

## Security Model

| Scenario | Vault key on disk? | Plaintext files? | What attacker gets |
|----------|-------------------|-------------------|-------------------|
| Working copy clone | Yes (VAULT-KEY) | Yes | Everything |
| Bare clone | No | No | Encrypted blobs + tree.json (paths only) |
| Bare zip on S3 | No | No | Encrypted blobs (need vault key) |
| Credential store | Encrypted in vaults.enc | N/A | Need passphrase to decrypt |

## Open Questions

1. **tree.json in bare mode** — Currently plaintext (shows file paths/sizes). Should bare vaults encrypt it too?
2. **Lambda layer packaging** — Should `pack` produce a Lambda-layer-compatible zip? (add prefix?)
3. **Bare vault pull** — Should `pull` work on bare vaults? (Update objects without extracting files)
4. **Credential store location** — One global `~/.sg-send/vaults.enc`, or also support project-local?
5. **Auto-add on clone** — Should `sg-send-cli clone` auto-offer to save the vault key to the credential store?

---

## Implementation Priority

1. **`~/.sg-send/` + credential store** — Immediate UX win, builds on Secrets__Store
2. **`vault add/list/remove/show` commands** — CLI for credential management
3. **Bare clone mode** — `clone --bare`, `checkout`, `clean`
4. **Zip container** — `pack`, `clone --bare --zip`, `--from` flag
5. **Session agent** — Future (like ssh-agent)
