# Architecture Review — Cloneless Vault Access

**Date:** 11 March 2026
**Status:** Design proposal
**Requested by:** Dinis Cruz
**Priority:** High — blocking Lambda/serverless adoption

---

## Problem Statement

The current vault workflow requires a **full clone** before any file can be read:

```
sg-send-cli clone {key}:s09yqbpj    # 13.3s for 25 files / 248 KB
```

This is problematic for:
1. **Lambda cold starts** — 13s added to init time just to read one config file
2. **Deploy service** — Only needs `EC2.json` but downloads entire vault
3. **CI/CD pipelines** — Fetch one secret/config, not 25 files
4. **Browse/search** — Cannot list vault contents without full clone

### Why It's Slow

The clone does 3 categories of work:
1. **Metadata** (2 API calls): Download + decrypt settings.json and tree.json
2. **Files** (N API calls): Download + decrypt each file sequentially
3. **Local state** (CPU): Build object store, commit chain, write .sg_vault/

For the `s09yqbpj` vault: 2 metadata + 25 file downloads = **27 sequential HTTP requests**.

---

## Design: Vault__Remote

A new class that operates **entirely against the remote API** — no local .sg_vault directory, no object store, no commit chain. Pure stateless read operations.

### Core Principle

The tree.json already contains a complete file manifest with file_ids. We can:
1. Download + decrypt tree.json (1 API call)
2. Use the tree to list/browse files (zero additional calls)
3. Download + decrypt a specific file by its file_id (1 API call per file)

**Total for "read one file": 2 API calls** (vs 27 for clone)

### Proposed API

```python
class Vault__Remote(Type_Safe):
    crypto : Vault__Crypto
    api    : Vault__API

    def list_files(self, vault_key: str) -> list[dict]:
        """List all files in the vault with path, size, file_id.
        Cost: 2 API calls (settings + tree). No local state."""

    def read_file(self, vault_key: str, file_path: str) -> bytes:
        """Read a single file by path.
        Cost: 3 API calls (settings + tree + file). No local state."""

    def read_file_by_id(self, vault_key: str, file_id: str) -> bytes:
        """Read a file when you already know its file_id.
        Cost: 1 API call. Fastest path for known files."""

    def get_tree(self, vault_key: str) -> dict:
        """Get the full tree metadata (version, updated, file manifest).
        Cost: 2 API calls. Returns structured tree with folder hierarchy."""

    def save_file(self, vault_key: str, file_path: str, dest: str) -> str:
        """Download and save a single file to local disk.
        Cost: 3 API calls. Returns path to saved file."""
```

### CLI Commands

```bash
# List files in a vault (no clone needed)
sg-send-cli ls {key}:s09yqbpj

# Read a file to stdout (pipe-friendly)
sg-send-cli cat {key}:s09yqbpj path/to/EC2.json

# Download a single file
sg-send-cli get {key}:s09yqbpj path/to/EC2.json ./local-copy.json

# Show vault info without cloning
sg-send-cli info {key}:s09yqbpj
```

### Performance Comparison

| Operation | Clone workflow | Cloneless workflow |
|-----------|---------------|-------------------|
| List files | 27 API calls + 13s | 2 API calls + ~0.5s |
| Read 1 file | 27 API calls + 13s | 3 API calls + ~0.8s |
| Read 1 file (cached tree) | N/A | 1 API call + ~0.3s |
| Lambda cold start | 13s+ | <1s |

## Architecture Decisions

### 1. Stateless by Default

Vault__Remote holds no local state. Each call is self-contained:
- Derive keys from vault_key
- Fetch metadata from API
- Decrypt and return

This makes it safe for Lambda, containers, and short-lived processes.

### 2. Optional Tree Caching

For repeated access within a session, callers can cache the tree:

```python
remote = Vault__Remote(crypto=crypto, api=api)
tree   = remote.get_tree(vault_key)              # 2 API calls

# Subsequent reads skip the tree fetch
data1 = remote.read_file_by_id(vault_key, tree['files']['EC2.json']['file_id'])  # 1 call
data2 = remote.read_file_by_id(vault_key, tree['files']['deploy.json']['file_id'])  # 1 call
```

The cache is the caller's responsibility — Vault__Remote stays stateless.

### 3. No Write Support Initially

Phase 1 is read-only:
- `list_files`, `read_file`, `save_file`, `get_tree`

Phase 2 (future) could add:
- `write_file(vault_key, file_path, data)` — upload one file, update tree
- `delete_file(vault_key, file_path)` — remove one file, update tree

Write support requires updating tree.json atomically, which introduces consistency concerns (what if two writers update simultaneously?). Defer this.

### 4. Reuse Existing Crypto and API

Vault__Remote uses the same `Vault__Crypto.derive_keys_from_vault_key()` and `Vault__API.read()` that clone already uses. No new API endpoints needed. No new crypto paths.

### 5. Relationship to Clone

Clone is not replaced — it's still the right choice when you need:
- Full local copy for editing
- Push/pull workflow
- Offline access
- Object store / commit history

Cloneless is for: "I know what file I want, give it to me fast."

---

## Security Considerations

- Same encryption model — all files decrypted client-side with read_key
- Vault key is still the single access credential
- No additional attack surface (uses existing API endpoints)
- Tree metadata reveals file paths and sizes (same as current clone behavior)

---

## Open Questions

1. **Tree caching in CLI** — Should `sg-send-cli ls` cache tree.json in /tmp for subsequent `cat`/`get` calls? Or always refetch?
2. **Python API packaging** — Should Vault__Remote be importable as a library for Lambda use? (Currently CLI-only entry point)
3. **Batch download** — Should `get` support glob patterns? e.g., `sg-send-cli get key configs/*.json ./local/`
4. **Version pinning** — Should cloneless reads support reading a specific vault version? (tree.json only has latest)

---

## Implementation Estimate

- **Vault__Remote class**: Small — reuses existing crypto/api, ~80 lines
- **CLI commands** (ls, cat, get, info): ~60 lines in CLI__Vault
- **Tests**: ~15 new tests (list, read, save, error cases)
- **Total**: Modest change, high impact
