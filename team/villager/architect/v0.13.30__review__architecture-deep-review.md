## Part 1: What Is Good — Problems This Architecture Solves

### 1.1 Eliminates `_flatten_tree()` / `_build_tree_json()` (Dev)

The current `Vault__Sync.py` (lines 340-410) maintains a nested JSON tree with a flat
file translation layer (`_flatten_tree`, `_build_tree_json`). This is the most fragile
part of the codebase — it requires keeping two representations in sync, and the tests
we just wrote (WP-7) revealed the complexity of testing these helpers.

The sub-tree model (Section 4, arch spec) eliminates this entirely. The vault IS the
graph. One tree object per directory level, content-addressed. No translation.

**Impact:** Deletes ~100 lines of the most bug-prone sync code.

### 1.2 Fixes the Same-Size Edit Detection Bug (Dev/QA)

The arch spec (Section 4.1) explicitly calls out:

> `content_hash` fixes the size-only change detection bug in the current codebase
> (line 359 of `Vault__Sync.py`). Same-size edits are now detected reliably.

Our test suite currently checks `status` against file size only (confirmed in
`test_Vault__Sync__Helpers.py::test_status__detects_added_file`). The architecture
adds `SHA256(plaintext)[:12]` as an encrypted field in tree entries. This is the
correct fix — size is not a reliable change detector.

### 1.3 Solves the "Who Signed This?" Problem (AppSec)

The current `Schema__Object_Commit` has no `signature` or `branch_id` field. There
is zero provenance. The PKI system (`PKI__Crypto`, `PKI__Key_Store`) exists but is
completely disconnected from the commit model.

The architecture connects them: every commit is signed by the branch key that created
it. The `parents[]` list enables proper merge commit provenance. This makes the
existing PKI infrastructure useful instead of ornamental.

### 1.4 Named Branch = Clean Integration Point (Architect)

The current codebase has a single ref (`refs/head`) which means push/pull conflicts
are unresolvable — whoever pushes last wins, and there's no merge. The three-layer
model (remote → local named branch → clone work branch) is the correct architecture.

Key insight: **conflicts only on clone branch, never on named branch.** This was
found during simulation (Gap #5) and is the single most important design decision
in the spec. It means the integration point is always clean, which enables the
collaborative resolution flow (`--branch-only` push).

### 1.5 `bare/` + `local/` Separation (Dev/DevOps)

The current `.sg_vault/` mixes portable vault content with local-only config
(the vault key file, the object store, refs). The `bare/` + `local/` separation
makes the portable unit explicit:

- `bare/` = the vault. Syncs. Zips. Backs up.
- `local/` = your device. Keys, tokens, tracking.

This eliminates the implicit "what do we sync?" question that currently lives in
the `Vault__Sync.push()` logic.

### 1.6 Batch API Reduces Round Trips (DevOps)

Current push does N individual API calls (one per file). The batch endpoint
(`POST /api/vault/batch/{vault_id}`) with Lambda doing direct S3 PUTs reduces
this to 1 GET + 1 POST. For a 25-file push, that's 27 round trips → 2.

The `write-if-match` optimistic locking on refs is the right pattern for
concurrent pushers.

### 1.7 Change Packs Enable Write-Only Append (Architect)

The `bare/pending/` namespace with scoped write-only tokens is architecturally
elegant. External systems (Lambdas, agents, IoT devices) can write to a vault
without ever reading it. The vault becomes a secure append-only log for external
producers. This was not possible with the current architecture.

### 1.8 Zero-Knowledge File Naming (AppSec)

Our AppSec tests (`test_AppSec__Vault_Security.py`) already verified that file
contents are encrypted in the object store. But the current tree structure leaks
filenames in plaintext (they're in the JSON tree entries before encryption).

The architecture encrypts filenames inside tree entries (`name_enc`). The server
sees only `obj-{hash}` — can't distinguish blobs from trees from commits. This
is a genuine zero-knowledge improvement.

### 1.9 Simulation as Test Oracle (QA)

The QA predictions table in the simulation document provides exact file counts
at every step of every scenario. These are directly translatable to integration
tests. QA doesn't have to reverse-engineer expected behavior — it's specified.

## Part 2: Issues Found — Stress Testing the Architecture

### ISSUE-1: `Schema__Object_Commit.parent` → `parents[]` Migration (HIGH)

**Section 4.1** changes `parent: str` to `parents: list[str]`.

The spec says: "Backward compatibility: read old single-parent format by treating
`parent` as `parents: [parent]`."

**Problem:** The current `Schema__Object_Commit` uses `parent: Safe_Str__Object_Id`.
The Type_Safe framework doesn't support union types or migration adapters. The
`from_json()` round-trip invariant (CLAUDE.md rule #6) will break if old commits
have `parent` and new commits have `parents`.

**Recommendation:** The schema should support BOTH fields during migration. Add
`parents: list[Safe_Str__Object_Id]` alongside `parent`. The `from_json()` method
reads `parents` if present, falls back to `[parent]` if only `parent` exists.
Add a property method `parent_ids` that normalizes both forms. Document the
migration phase explicitly.

### ISSUE-2: Branch Index Staleness During Push Race (MEDIUM)

**Section 5.4** states the branch index is "eventually consistent — may be stale
between syncs."

**Problem:** During push step 7, the batch writes include an updated branch index.
But if two users push concurrently, the last writer wins on the index, potentially
dropping the other's branch entry. Unlike refs (which use `write-if-match`), the
index has no concurrency protection.

**Recommendation:** Either (a) apply `write-if-match` to the branch index too,
or (b) explicitly state that clone registration is the only time the index is
written and concurrent index writes are resolved by GC on the next sync. Option
(b) is simpler and consistent with the eventual consistency model, but it should
be spelled out.

### ISSUE-3: Init Commit Signed by Clone Key, But Clone Key Has No Published Public Key Yet (MEDIUM)

**Section 6.1, Step 10:** The init commit is signed by the clone branch key.
The public key is stored in `bare/keys/key-{id}`.

**Problem:** At init time, there is no remote. The public key is stored locally.
When a second user clones, they receive the public key and can verify. But between
init and first push, the init commit's signature is unverifiable by anyone except
the creator. This is fine — but the spec doesn't state this explicitly.

**Also:** If the init device is lost before the first push, the clone branch
private key is lost. The init commit becomes permanently unverifiable (the public
key is in `bare/keys/` but the signature can't be validated against a trusted
source). Is this acceptable?

**Recommendation:** Add a note: "The init commit's signature is verifiable after
the first push (when bare/keys/ is synced to the server). Before that, it is
self-attested only."

### ISSUE-4: No `content_hash` Verification on Pull (MEDIUM)

**Section 4.1:** Tree entries include `content_hash: SHA256(plaintext)[:12]`.

**Problem:** The spec describes using `content_hash` for change detection during
commit (Section 6.2, step 3). But it doesn't specify whether `content_hash` is
verified during pull/checkout. If someone with the vault key creates a commit
with a deliberately wrong `content_hash`, the file still decrypts correctly —
the hash is advisory, not enforced.

**Recommendation:** Specify that `content_hash` is verified during checkout:
after decrypting a blob, compute `SHA256(plaintext)[:12]` and compare to the
tree entry. If mismatch, raise a warning (not an error — the content is still
usable, but integrity is degraded). This gives us tamper detection for free.

### ISSUE-5: GC Prune Before All Replicas Sync (LOW)

**Section 11.5, Step 8:** GC prunes processed pending objects.

**Problem:** If Device A runs GC and prunes pending objects, but Device B hasn't
synced yet, Device B won't see the pending objects — only the resulting commits.
This is correct behavior (the commits contain everything). But if Device B has
a stale branch index that still lists pending counts, it may try to enumerate
`bare/pending/` and find nothing, which could be confusing.

**Recommendation:** GC should update the pending counts index (idx-{pending_counts})
as part of step 6 (REBUILD). The spec mentions this but doesn't explicitly tie
the prune step to the index update. Make the ordering explicit: index update
BEFORE prune.

### ISSUE-6: `Vault__API.read()` Has No Auth — List Endpoint Exposure (LOW)

**Section 12.3:** The list endpoint is "read-unauthenticated (same as read —
the file_ids are opaque, and content is encrypted)."

**Problem from codebase:** The current `Vault__API.read()` (line 27) has no
authentication. This is consistent with the spec. However, the list endpoint
reveals **counts** — an attacker can enumerate `bare/pending/` to see how many
change packs exist, or `bare/branches/` to see how many collaborators there are.
This is metadata the spec says the server already knows (Section 3.4), but the
list endpoint makes it queryable by anyone, not just the server.

**Recommendation:** Consider rate-limiting or token-gating the list endpoint.
Or explicitly document that file counts are considered acceptable metadata
exposure (consistent with the server's existing knowledge).

### ISSUE-7: Missing Error Handling in Batch Endpoint Partial Failure (MEDIUM)

**Section 12.2:** "Atomicity: best-effort. Immutable objects can be written in
any order."

**Problem:** The spec says if the ref `write-if-match` fails, "the written
objects are harmless orphans that GC can clean up." But it doesn't specify:
- What HTTP status code does the batch endpoint return on partial failure?
- Does the client retry the entire batch, or just the ref update?
- What if one of the immutable object writes fails (S3 error)?

**Recommendation:** Define the batch response schema:
```json
{
  "results": [
    { "file_id": "...", "status": "ok" },
    { "file_id": "...", "status": "conflict", "current": "..." }
  ],
  "overall": "partial"
}
```
The client retries from step 2 (fetch) on any ref conflict, and retries individual
object writes on transient errors.

## Part 4: Implementation Risk Assessment

### High Risk

1. **Three-way merge at sub-tree level** — ~300-500 LOC, correctness-critical.
   The merge table (Section 8.3) has 11 cases. Each must be tested with real
   encrypted objects. This is the hardest new code.

2. **`parent` → `parents[]` migration** — Affects every existing test that creates
   commits. Must maintain backward compatibility during transition.

3. **Batch API server-side implementation** — Requires Lambda changes, S3 direct
   calls, `write-if-match` semantics. Not a CLI-only change.

### Medium Risk

4. **Commit signing enforcement** — "No unsigned commits" means every test that
   creates a commit must now generate or mock a key pair. This affects ~40 tests.

5. **`bare/` folder restructure** — Current `.sg_vault/objects/` → `bare/data/`.
   Affects `Vault__Object_Store.object_path()`, all tests that use temp dirs.

### Low Risk

6. **New CLI commands** — `commit`, `merge`, `branches`, `remote` — standard
   CLI expansion following existing patterns.

7. **Uint timestamps** — Mechanical replacement of ISO strings with uint ms.
   `Safe_Str__ISO_Timestamp` → `Safe_UInt__Timestamp`.

## Verdict

**GO** — The architecture is sound, well-validated by simulation, and solves real
problems we've observed in the current codebase. The seven issues above should be
resolved before implementation begins, but none are architectural blockers.

The simulation-driven methodology (case study document) is genuinely valuable.
The 14 gaps found during simulation would have cost weeks to discover during
implementation. We recommend this methodology for all future architecture work.

**Priority order for issue resolution:**
1. ISSUE-1 (parent → parents migration) — blocks Phase A
2. ISSUE-7 (batch error handling) — blocks Phase C
3. ISSUE-2 (index staleness) — clarify before Phase D
4. ISSUE-4 (content_hash verification) — design decision for Phase A
5. ISSUE-3 (init commit verifiability) — documentation, non-blocking
6. ISSUE-6 (list endpoint exposure) — AppSec review, non-blocking
7. ISSUE-5 (GC prune ordering) — clarify before Phase F
