# Architecture Brief: Git-Like Encrypted Object Store

**Version:** v0.11.1
**Date:** 4 March 2026
**Roles:** Architect + Dev
**Status:** Architecture brief — **draft for review**

---

## Executive Summary

The current sg-send-cli is a **file-sync tool** — it pushes and pulls encrypted files between a local directory and the SG/Send server. Every push overwrites the tree in place. There is no history, no conflict detection, and no way to recover a previous state.

This brief proposes evolving the system into a **git-like encrypted object store** — where every push creates an immutable commit, history is preserved, and conflicts are detectable. The core insight: the server is a dumb key-value blob store (PUT/GET/DELETE by vault_id + file_id). All intelligence lives in the client. **No server changes are required.**

---

## 1. Current State

### What Works Well (Keep As-Is)
- **Vault__Crypto** — PBKDF2 key derivation + AES-256-GCM encryption. Production-ready, cross-language tested. Zero changes needed.
- **Safe types framework** — 11 domain-specific types with regex validation. Extensible.
- **Type_Safe architecture** — Clean, no raw primitives, round-trip invariant enforced.
- **API security model** — Zero-knowledge preserved. Write key + access token for writes, no auth for reads.

### What's Limited (Must Change)
- **No history** — Tree is overwritten on every push. Previous states are lost permanently.
- **No conflict detection** — Two concurrent pushes silently clobber each other (last writer wins).
- **Size-only change detection** — `Vault__Sync` compares file sizes only (not content hashes). Two files of the same size but different content are not detected as modified.
- **Random file IDs** — User file IDs are `os.urandom(6).hex()`. Modified files get new random IDs; old blobs are orphaned on the server.
- **No deduplication** — Identical files are stored multiple times.

### Server Constraints (Cannot Change Without Server Work)
1. **No LIST endpoint** — Cannot enumerate objects in a vault
2. **No conditional writes** — No compare-and-swap, no ETags, no version checks
3. **No server-side ordering** — Server doesn't know about commits or history
4. **Write endpoint is idempotent overwrite** — Same file_id = same slot

These constraints shape the entire design: the client must track all state, and the server remains a dumb blob store.

---

## 2. Design Goals

1. **Immutable history** — Every push creates a commit object. Previous states are preserved and retrievable.
2. **Content integrity** — Every object (blob, tree, commit) has a verifiable identity derived from its encrypted content.
3. **Conflict detection** — Concurrent pushes are detected via parent-commit lineage rather than silently overwritten.
4. **Efficient sync** — Only transfer objects the other side doesn't have.
5. **No server changes** — The server stays a dumb key-value store (PUT/GET/DELETE).
6. **Zero-knowledge preserved** — The server sees only opaque encrypted blobs and their IDs.

## 3. Object Model

### 3.1 Object Types

Three object types, mirroring git's model but encrypted:

| Type | Plaintext Contents (before encryption) | Identity |
|------|----------------------------------------|----------|
| **blob** | Raw file bytes | SHA-256 of ciphertext (post-encryption) |
| **tree** | JSON mapping: `{path -> {blob_id, size}}` | SHA-256 of ciphertext |
| **commit** | JSON: `{parent, tree_id, timestamp, message, version}` | SHA-256 of ciphertext |

**Why SHA-256 of ciphertext, not plaintext?**
1. The server never sees plaintext. Content-addressing by plaintext hash would leak information (identical files across vaults would have the same ID).
2. AES-GCM with random IVs means identical plaintext produces different ciphertext. This sacrifices deduplication but preserves security.
3. The ciphertext is what gets stored. Its hash is what can be verified for integrity.

**Object ID format:** First 12 hex characters of SHA-256(ciphertext). This gives 48 bits — birthday bound at ~16.7M objects, more than sufficient for vault-scale usage. Crucially, this matches the existing `Safe_Str__File_Id` format (12 lowercase hex chars), so **no server-side validation changes are needed**. If collision risk becomes a concern, the ID length can be extended later.

### 3.2 Object Wire Format

Every object stored on the server:

```
[1 byte type tag][12 bytes IV][ciphertext + 16-byte GCM tag]
```

Type tags:
- `0x01` = blob
- `0x02` = tree
- `0x03` = commit

**Decision: Type tag inside the encrypted envelope.** The server should not know whether an object is a blob, tree, or commit. This preserves zero-knowledge. The client always knows what type it expects when fetching (it follows references from commits to trees to blobs).

### 3.3 Blob Object

Plaintext:
```
[raw file bytes]
```

Encrypted and stored at `object_id = sha256(ciphertext)[:24]`.

### 3.4 Tree Object

Plaintext (JSON):
```json
{
    "entries": {
        "docs/readme.md": {
            "blob_id": "a1b2c3d4e5f6",
            "size": 1024
        },
        "src/main.py": {
            "blob_id": "b2c3d4e5f6a1",
            "size": 2048
        }
    }
}
```

**Key change from current design:** The tree uses flat paths (like `git ls-tree -r`) instead of nested folder objects. This simplifies diffing and merging. The current nested `{type: "folder", children: {...}}` structure adds complexity without benefit for a CLI tool.

### 3.5 Commit Object

Plaintext (JSON):
```json
{
    "parent": "c3d4e5f6a1b2",
    "tree_id": "d4e5f6a1b2c3",
    "timestamp": "2026-03-04T12:00:00.000Z",
    "message": "Add documentation files",
    "version": 5
}
```

- `parent` is `null` for the initial commit
- `version` is an auto-incrementing integer for human readability (commit 1, 2, 3...)
- `tree_id` references the tree object for this commit's state
- `message` is optional (defaults to auto-generated summary like "Push: 2 added, 1 modified")

### 3.6 Ref (HEAD Pointer)

The ref is a mutable pointer stored at a new deterministic `ref_file_id`, derived via:

```
ref_file_id = HMAC-SHA256(read_key, "sg-vault-v1:file-id:ref:{vault_id}")[:12]
```

Its plaintext (before encryption) contains:

```json
{
    "commit_id": "e5f6a1b2c3d4",
    "version": 5
}
```

This is the **only mutable object** in the entire system (alongside settings). Everything else (blobs, trees, commits) is immutable and content-addressed. The ref is overwritten on each push to point to the latest commit.

**Why a new `ref_file_id` instead of reusing `tree_file_id`?** Clean separation during migration. The old `tree_file_id` slot may still contain a legacy tree on the server. A new derivation domain avoids collision and makes migration detection simple: if `refs/head.json` exists locally, the vault uses the object store; if not, it's legacy format.

## 4. Storage Layout

### Remote (Server)

```
transfers/vault/{vault_id}/
    {tree_file_id}/payload          ← HEAD pointer (mutable, overwritten)
    {settings_file_id}/payload      ← Settings (mutable, overwritten)
    {object_id_1}/payload           ← Blob (immutable)
    {object_id_2}/payload           ← Tree snapshot (immutable)
    {object_id_3}/payload           ← Commit (immutable)
    ...
```

All objects share the same flat namespace under the vault. The server doesn't know or care about object types.

### Local (`.sg_vault/`)

```
.sg_vault/
    HEAD                    ← Vault key string (unchanged)
    settings.json           ← Decrypted settings (unchanged)
    token                   ← Saved access token (unchanged)
    refs/
        head                ← Current commit_id (24 hex chars)
    objects/                ← Local object cache (mirrors server)
        a1/b2c3d4e5f6...   ← Cached objects (2-char prefix dirs, like git)
    tree.json               ← Decrypted tree from current commit (working copy)
    commit_log.json         ← Local commit history cache (for offline `log`)
```

**Object cache:** Downloaded objects are cached locally in `.sg_vault/objects/`. On push, new objects are created locally first, then uploaded. On pull, missing objects are downloaded and cached. This enables:
- Offline `status` and `log` commands
- Efficient sync (only transfer objects the other side doesn't have)
- Integrity verification (recompute hash, compare to ID)

## 5. Command Flows

### `sg-vault push [directory]`

```
1. Read .sg_vault/HEAD → vault key → derive keys
2. Read .sg_vault/refs/head → current_commit_id
3. Read .sg_vault/tree.json → old_tree (from last known state)
4. Scan local directory → build new_tree (flat path → {size, sha256_of_content})
5. Diff old_tree vs new_tree → added, modified, deleted files

6. For each added/modified file:
   a. Read file bytes → encrypt with read_key → ciphertext
   b. blob_id = sha256(ciphertext)[:24]
   c. Store in local object cache
   d. Upload: PUT /api/vault/write/{vault_id}/{blob_id}

7. Create tree object:
   a. Build tree JSON with {path → {blob_id, size}} for all files
   b. Encrypt → tree_ciphertext
   c. tree_id = sha256(tree_ciphertext)[:24]
   d. Upload tree object

8. Create commit object:
   a. Build commit JSON: {parent: current_commit_id, tree_id, timestamp, message, version}
   b. Encrypt → commit_ciphertext
   c. commit_id = sha256(commit_ciphertext)[:24]
   d. Upload commit object

9. Conflict check:
   a. Download current HEAD pointer from server
   b. If server HEAD commit_id != current_commit_id → CONFLICT
      - "Remote has changed since your last pull. Run `sg-vault pull` first."
      - Abort push (do not update HEAD)
   c. If match → safe to update

10. Update HEAD pointer:
    a. Build HEAD JSON: {commit_id, version}
    b. Encrypt → PUT /api/vault/write/{vault_id}/{tree_file_id}

11. Update local state:
    a. Write commit_id to .sg_vault/refs/head
    b. Write new tree to .sg_vault/tree.json
    c. Append to .sg_vault/commit_log.json
    d. Print summary: "Committed v5: 2 added, 1 modified, 0 deleted"

12. For each deleted file:
    (Optional) DELETE /api/vault/delete/{vault_id}/{old_blob_id}
    Note: Orphaned blobs don't break anything. GC can be deferred.
```

**Conflict detection (step 9):** This is an optimistic concurrency check. The push uploads all objects first (they're immutable and harmless), then checks the HEAD pointer before updating it. If another client pushed in between, the HEAD has moved and we abort. The uploaded objects are not wasted — they'll be referenced by the eventual commit after the user pulls and retries.

**Race condition:** There's a TOCTOU gap between checking HEAD (step 9) and updating it (step 10). Without server-side compare-and-swap, this can't be eliminated entirely. However, the window is tiny (one HTTP round trip) and the worst case is two commits with the same parent — which is detectable on the next pull.

### `sg-vault pull [directory]`

```
1. Read .sg_vault/HEAD → vault key → derive keys
2. Read .sg_vault/refs/head → local_commit_id
3. Download HEAD pointer: GET /api/vault/read/{vault_id}/{tree_file_id}
   → decrypt → {commit_id: remote_commit_id, version}

4. If remote_commit_id == local_commit_id → "Already up to date."

5. Download commit chain:
   a. Fetch remote_commit_id → decrypt → commit object
   b. Walk parent pointers until reaching local_commit_id (or null)
   c. This gives us the list of new commits

6. Download the latest commit's tree:
   a. Fetch tree_id from latest commit → decrypt → remote tree

7. Diff local tree vs remote tree:
   → added, modified, deleted files

8. For each added/modified file:
   a. Download blob: GET /api/vault/read/{vault_id}/{blob_id}
   b. Decrypt → write to local directory
   c. Cache in .sg_vault/objects/

9. For each deleted file:
   a. Remove from local directory

10. Update local state:
    a. Write remote_commit_id to .sg_vault/refs/head
    b. Write remote tree to .sg_vault/tree.json
    c. Update .sg_vault/commit_log.json with new commits
    d. Print summary
```

### `sg-vault clone <vault-key> [directory]`

```
1. Parse vault key → passphrase, vault_id
2. Derive keys: read_key, write_key, tree_file_id, settings_file_id
3. Download HEAD pointer → decrypt → {commit_id, version}
4. Download commit → decrypt → {tree_id, ...}
5. Download tree → decrypt → file list
6. Download settings → decrypt
7. Create local directory
8. Download and decrypt all blobs → write to files
9. Create .sg_vault/ with HEAD, refs/head, tree.json, settings.json, token
10. Cache all downloaded objects in .sg_vault/objects/
```

### `sg-vault log [--oneline] [--limit N]`

```
1. Read .sg_vault/refs/head → current commit_id
2. Read .sg_vault/commit_log.json (if available, use cached)
3. Otherwise, walk commit chain from HEAD:
   a. Fetch commit → decrypt → show: version, timestamp, message, commit_id
   b. Follow parent pointer → repeat
   c. Stop at null parent or --limit

Output:
  commit e5f6a1b2c3d4 (HEAD)
  Version: 5
  Date:    2026-03-04T12:00:00Z
  Message: Add documentation files

  commit c3d4e5f6a1b2
  Version: 4
  Date:    2026-03-03T18:30:00Z
  Message: Push: 1 added, 0 modified, 0 deleted
```

### `sg-vault checkout <version>`

```
1. Walk commit chain to find commit with matching version number
2. Download that commit's tree → decrypt
3. Diff current working directory against target tree
4. Apply changes (add/modify/delete files)
5. Update .sg_vault/refs/head to target commit_id
6. Update .sg_vault/tree.json
7. Print: "Checked out version 3 (commit a1b2c3d4e5f6)"

Note: This is a "detached HEAD" state if not at the latest commit.
Subsequent pushes from a detached HEAD create a fork.
```

### `sg-vault status` (unchanged)

Same as current — compare local directory against `.sg_vault/tree.json`.

### `sg-vault remote-status` (enhanced)

Now shows commit divergence:
```
Local version:  5 (commit e5f6a1b2c3d4)
Remote version: 7 (commit f6a1b2c3d4e5)

Remote is 2 commits ahead:
  v7: Update API docs
  v6: Fix typo in readme

Local changes (not yet pushed):
  + new-file.txt
  ~ modified-file.txt
```

## 6. Key Design Decisions

### D1: Deduplication — Accept No Dedup

AES-GCM with random IVs means identical plaintext produces different ciphertext. Two options:

| Option | Approach | Pros | Cons |
|--------|----------|------|------|
| A. Accept no dedup | Random IV per encryption | Secure, simple | Duplicate storage |
| B. Deterministic IV | HMAC(read_key, plaintext_hash) as IV | Dedup works | Leaks "these blobs have same content" to server |

**Decision: Option A.** Security over storage efficiency. Dedup is a nice-to-have, not a requirement. The zero-knowledge property is paramount.

### D2: Object ID Length — 12 Hex Chars (Keep Existing Format)

Current file_ids are 12 chars (48 bits). The Architect recommends keeping this format.

| Length | Bits | Collision at | Recommendation |
|--------|------|--------------|----------------|
| 12 chars | 48 bits | ~16.7M objects | Sufficient for vault-scale. Matches server validation. |
| 24 chars | 96 bits | ~2^48 objects | More headroom, but requires server validation changes |
| 64 chars | 256 bits | Full SHA-256 | Overkill |

**Decision: 12 hex chars.** Matches existing `Safe_Str__File_Id` format, so no server-side changes needed. A vault with thousands of files and hundreds of commits is nowhere near the 16.7M collision bound. Can be extended later if needed.

### D3: Tree Format — Flat Paths

Current tree uses nested folder objects. The git-like model uses flat paths:

```json
// Current (nested)
{"type": "folder", "children": {"docs": {"type": "folder", "children": {"readme.md": {"type": "file", ...}}}}}

// Proposed (flat)
{"entries": {"docs/readme.md": {"blob_id": "...", "size": 1024}}}
```

**Decision: Flat paths.** Simpler to diff, merge, and serialize. Folders are implicit from paths. The nested format adds complexity without benefit for a CLI tool.

### D4: Garbage Collection — Deferred

When files are modified, old blob objects become unreferenced. Options:
- **Eager delete:** DELETE old blobs on push
- **Deferred GC:** Leave orphans, periodically clean up

**Decision: Deferred GC.** Orphaned blobs cost storage but don't break anything. A future `sg-vault gc` command can walk the commit graph, identify referenced objects, and DELETE unreferenced ones. This keeps push fast and simple.

### D5: Conflict Strategy — Detect, Don't Auto-Merge

Git does 3-way merges. For encrypted vaults:
- The server has no LIST endpoint, so we can't compare object sets server-side
- Files are encrypted, so server-side merge is impossible
- Most vault use cases are single-user or low-contention

**Decision: Detect conflicts, require manual resolution.** On push, check if remote HEAD matches local HEAD's parent. If not, abort with "pull first." On pull with local changes, warn "you have uncommitted local changes" (like `git stash` requirement).

Future enhancement: implement client-side 3-way merge for text files.

### D6: HEAD Pointer vs. Commit Chain

Two approaches to finding history:

| Approach | How it works | Pros | Cons |
|----------|-------------|------|------|
| Single HEAD | HEAD points to latest commit. Walk parent chain. | Simple, no server changes | Must walk chain to find history |
| Commit log file | Store commit list as a separate mutable file | Fast log lookups | Another mutable file to manage |

**Decision: Both.** HEAD is the source of truth (always points to latest commit). A local `commit_log.json` caches the history for fast `log` commands without network access. The commit chain is the authoritative history; the log cache is rebuilt on clone.

## 10. Migration Path: Legacy → Object Store

**Detection:** If `.sg_vault/refs/head.json` does not exist, the vault is in legacy format.

**Automatic migration on first push with new client:**

1. Read `.sg_vault/tree.json` → old nested tree
2. Flatten into list of `(file_path, file_id, size)` entries
3. **Adopt existing random file_ids as blob object_ids** (no re-upload needed)
4. Build `Schema__Object_Tree` with the original file_ids as blob references
5. Store tree object → `tree_id`
6. Create initial commit: `{tree_id, parent: null, version: current_version, message: "Migrated from legacy format"}`
7. Store commit object → `commit_id`
8. Create ref: `{commit_id, version}` → upload to `ref_file_id`
9. Write `.sg_vault/refs/head.json` and `.sg_vault/log.json`
10. Continue with normal push flow

**Key design choice:** Legacy blob file_ids are adopted as-is. They are not content-addressed, but that is a cosmetic inconsistency — not a functional problem. New uploads going forward use proper content-addressed IDs. The migration is **additive** (no old data deleted), so rollback is safe.

## 11. Open Questions for Review

1. **Should we support branches?** Git's full branching model adds significant complexity. For v0.11.1, a single linear history (like `git` with only `main`) may be sufficient. Branches can be added later without breaking changes.

2. **Commit message UX:** Should push require a message (like `git commit -m`)? Or auto-generate one (like "Push: 2 added, 1 modified")? Recommendation: auto-generate by default, optional `--message` flag.

3. **Object ID in API paths:** Resolved — keeping 12-char hex format to match existing `Safe_Str__File_Id` / `Safe_Str__Transfer_Id` server validation. No server changes needed.

4. **Local-only commits?** Should there be a `sg-vault commit` (local only) separate from `sg-vault push` (commit + upload)? Git separates these. For simplicity, v0.11.1 could combine them (push = commit + upload).

5. **Maximum commit chain length:** Without a LIST endpoint, walking a long commit chain requires N sequential HTTP requests (one per commit). At 100+ commits, this becomes slow. Options: (a) cache aggressively locally, (b) add a commit-log file on the server, (c) add a LIST endpoint.

## 12. Security Considerations

### What's Preserved
- Zero-knowledge: Server sees only encrypted blobs and 12-char hex IDs
- Write authorization: Same write_key + access_token model
- Encryption: Same AES-256-GCM with random IVs

### What's New
- **Object IDs leak content identity:** SHA-256(ciphertext) means the server can tell if two vaults have the same encrypted blob. Since IVs are random, this only happens if the exact same ciphertext is reused (which shouldn't happen in practice).
- **Commit chain length leaks activity frequency:** The number of objects in a vault roughly correlates with activity. This is already true with the current design (more files = more objects).
- **Type tags are hidden:** Object types are inside the encrypted envelope. The server can't distinguish blobs from trees from commits.

### No Degradation
The security model is equivalent to the current design. The main change is from mutable objects (overwrite tree in place) to immutable objects (append new trees, update HEAD pointer). This doesn't expose additional information to the server.
