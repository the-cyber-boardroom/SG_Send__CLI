## 1. The Core Challenge

We want a system where:

- Multiple users and agents can collaborate on a shared vault
- Every participant has **cryptographic identity** (not just a username)
- The system is **zero-knowledge** end-to-end — the server is a dumb encrypted blob store
- High-performance paths are **single HTTP requests** where possible
- Consistency is **eventual** — conflicts are resolved explicitly, not prevented by locking
- The architecture works equally well for humans at a laptop, AI agents in Lambda, and CI/CD pipelines

This is a harder problem than most collaborative systems because we can't use the server as a coordinator — the server can't read anything.

## 2. Branch Per Clone — The Foundation

### Why this is the right primitive

Every clone creates a new branch with its own key pair. This is not just a UX convenience — it's a **security and architecture invariant**:

```
clone() → generate EC P-256 key pair → branch_id = fingerprint(public_key)
```

| Property | What it gives us |
|----------|-----------------|
| **Provenance** | Every commit traces to a branch, every branch traces to a key pair, every key pair traces to a device/session/agent |
| **Isolation** | You can only write to your own branch — no accidental overwrites |
| **Push always succeeds** | Your branch is yours alone. No remote conflicts at push time. |
| **Anonymous by default** | A branch key pair proves "same entity as last time" without revealing who that entity is |
| **Agent-native** | An AI agent gets the same first-class branch identity as a human |

### Key pair model — public/private split

**Public keys always live in the vault** — pushed to `keys/public/{branch_id}.pub` and listed in the vault's key registry. Anyone with the vault read key can see all active branch public keys and verify any commit's signature.

**Two kinds of private keys — different rules for each:**

| Key type | Can leave device? | Where stored | Lost key consequence |
|----------|-----------------|--------------|---------------------|
| **User's main key pair** | Yes — intended for multi-device | `~/.sg-send/`, OS keychain, browser storage, another vault | User re-imports from backup |
| **Branch private key** (clone-specific) | No — tied to this specific clone/device/session | Local only, never in vault | Branch becomes **permanently locked** |
| **Named branch private key** | Vault-stored (double-encrypted) | `keys/private/{branch_id}.enc` in vault | Depends on encryption scheme — see Section 3 |

The **branch private key** is deliberately local-only because it is the provenance record for *this specific clone, device, and environment*. If the same key appeared on two devices, you could no longer be certain which device/agent created a given commit. That provenance guarantee is the whole point.

The **user's main key pair** (RSA-4096 encryption + ECDSA P-256 signing) can and should be available across devices — it represents the *person or entity*, not a specific clone. Carrying it across devices via encrypted export, another vault, or a key manager is normal and expected.

```
Branch A private key (on Device 1):  stays on Device 1 forever
  → Commits from Branch A = provably from Device 1's clone

User key pair (on Device 1):         can be imported on Device 2
  → Allows user to open named branches, decrypt vault keys, etc.
  → Does NOT allow creating commits that look like they came from Branch A
```

**Permanently locking a branch** is done simply by deleting the branch private key. No ceremony, no command needed — once the key is gone, the branch is cryptographically read-only. The branch history remains intact and verifiable. This is a normal, expected lifecycle event — not a failure case. In many workflows a branch is created, used for a task, and then locked when the task is complete.

### Tree hashing — Git-compatible chain integrity

Every tree object contains a hash of the current commit content + parent commit hash(es), exactly as Git does. This creates a tamper-evident chain:

```
commit_id = hash(tree_id | parent_commit_id | timestamp | message | branch_fingerprint | signature)
```

Tampering with any historical commit invalidates all subsequent commit IDs. The commit_id itself is the integrity proof — no separate Merkle tree needed.

For merge commits:
```
merge_commit_id = hash(tree_id | parent_A_id | parent_B_id | timestamp | message | branch_fingerprint | signature)
```

### The identity hierarchy

```
Vault
  └── Branch A  (key pair: Device 1 clone — branch private key local to Device 1)
  │     └── Commit 1  (signed by Branch A key, parent: none)
  │     └── Commit 2  (signed by Branch A key, parent: Commit 1)
  │     └── Commit 3  (signed by Branch A key, parent: Commit 2)
  │     [branch key deleted → Branch A is now permanently locked / read-only]
  │
  └── Branch B  (key pair: Lambda agent — branch private key in Lambda env)
  │     └── Commit 1  (signed by Branch B key)
  │     └── Commit 2  (signed by Branch B key)
  │
  └── Branch C  (key pair: same user, Device 2 — different clone, different key)
  │     └── Commit 1  (signed by Branch C key)
  │
  └── main  [named branch — merge-only, own key pair, private key in vault]
        └── Merge commit  (parents: Branch A commit 3, Branch B commit 2)
        └── Merge commit  (parents: Branch C commit 1, ...)
```

## 3. Named Branches — A New Kind of Protected Branch

### The concept

A **named branch** is a special branch type that:
- Has its own PKI identity (key pair), just like a user branch
- Can **only receive merges** — direct commits are rejected
- Is the authoritative integration point (e.g., `main`, `release`, `production`)

This is something Git doesn't have natively. Git's branch protection rules are server-side policy, bolted on after the fact and not cryptographically enforced. Here, the protection is **in the format**: a commit targeting a named branch without going through a merge is structurally invalid — it won't have the right parent signature chain.

### Why named branches need their own key pair

The key pair on a named branch serves a different purpose than on a user branch:

| User branch key | Named branch key |
|----------------|-----------------|
| Proves "I am the same entity as last time I committed" | Proves "this merge was performed by an authorised entity" |
| Private key lives on user's device | Private key lives in the vault (double-encrypted) or with a designated authority |
| Lost key = branch abandoned, clone again | Lost key = branch locked, recover from vault |

The **named branch private key is stored in the vault** — encrypted with the vault read_key (and optionally with additional user keys). Any entity that holds the vault key can access the named branch key and perform merges into it.

### Double-encrypted key storage

The vault maintains a key registry at `objects/keys/`:

```
objects/keys/
  ├── registry.json                       → {key_id: {branch_id, key_type, created_at, ...}} (encrypted)
  ├── public/
  │   ├── {branch_id}.pub                 → branch public key (readable with vault read_key)
  │   └── {named_branch_id}.pub           → named branch public key (readable with vault read_key)
  └── private/
      └── {named_branch_id}.enc           → named branch private key, double-encrypted:
                                            outer: AES-256-GCM with vault read_key
                                            inner: optionally also encrypted with specific user key(s)
```

**The double-encryption pattern:**
- **Outer layer (vault read_key):** Anyone with the vault key can decrypt the outer layer. This is the "single vault key" scenario — transparent, simple, works for most cases.
- **Inner layer (user key, optional):** A specific user's public key can be used to wrap the inner layer. Only that user's private key can unwrap it. This enables **locked branches** — a named branch that only a specific person (or quorum) can merge into.

### Locked branches — what Git doesn't have

This design gives us branch-level access control that is cryptographically enforced, not policy-enforced:

```
Scenario A (simple): One vault key, one named branch key
  → Anyone with the vault key can merge into main
  → Transparent, works like current git flow

Scenario B (locked): Named branch key wrapped with team lead's public key
  → Only the team lead can merge into release
  → Even someone with the vault key cannot forge a merge
  → The lock is in the cryptography, not in server policy

Scenario C (quorum): Named branch key split (Shamir / multi-sig pattern)
  → N of M key holders must participate to merge
  → Future: not for initial implementation, but the design supports it
```

## 4. Change Packs — The Performance Primitive

### Why not normal commits for everything?

A standard commit requires: read HEAD → compute tree diff → write blobs → write tree → write commit → update ref. For performance-sensitive paths (Lambda billing events, telemetry, agent traces), this is too expensive. We want **one HTTP request per update**.

### What is a change pack?

A **change pack** is a self-contained bundle of one or more changes that has not yet been integrated into a branch's commit tree. It is:

- **Signed** by the creator's key (provenance preserved even before GC processes it)
- **Self-contained** — payload is inline (up to ~256KB) or a zip of multiple files
- **Deferred** — stored in `pending/` namespace, processed by GC into real commits later
- **Idempotent** — keyed by content hash, safe to submit multiple times

```
Normal commit path (multiple requests):
  POST blob_1, POST blob_2, POST tree, POST commit, PUT refs/heads/branch → 5+ requests

Change pack path (one request):
  POST pending/{branch_id}/{timestamp}_{random}  → 1 request
```

### The keyless creator insight

This is a significant capability: **a change pack can be created by an entity that has never cloned the vault and has no read access to it.**

All that's needed to create a valid change pack:
1. The **branch public key** (to address and encrypt the pack for that branch)
2. The **creator's own signing key** (to sign it — can be any key, not necessarily a vault key)
3. The **change pack schema** (the JSON format)

```
Scenario: External webhook receiver
  → Receives a payment event
  → Has never cloned the billing vault
  → Knows the billing branch's public key (published or shared out-of-band)
  → Creates and encrypts a change pack: { event: "payment", amount: 42.00, ... }
  → Signs it with its own service key
  → POSTs it to vault pending/{billing_branch_id}/
  → Done. One request. No vault clone needed.
```

The GC later drains `pending/billing_branch_id/`, verifies the signature, and integrates the change pack into the branch's commit tree. The vault owner can see *who* submitted each change pack (the submitter's public key), even if that submitter never had access to the vault contents.

This makes the vault a **write-only append target** for external systems — no read access required to contribute data.

### Change pack contents

A change pack can contain:
- A single JSON object (smallest case — one event)
- Multiple files (a zip payload)
- A mix of new files and metadata about what to do with them

```json
{
  "schema": "change_pack_v1",
  "branch_id": "{branch_id}",
  "created_at": "2026-03-12T14:23:00Z",    // UTC always
  "creator_key": "{submitter_public_key_fingerprint}",
  "signature": "{signature over payload_hash}",
  "payload_hash": "{sha256 of encrypted payload}",
  "payload": "{AES-256-GCM encrypted, inline or zip reference}"
}
```

Size limit: **~256KB encrypted inline**. Above this, use a zip reference (blob stored separately, change pack contains the blob_id).

### When to use change packs

| Use case | Why change packs? |
|----------|------------------|
| Lambda billing event | One event per invocation, sub-100ms budget |
| Agent trace/log entry | High frequency, unique per event, never conflict |
| External webhook receiver | Write-only contributor with no vault read access |
| CI artifact upload | Unique filename per build, no tree coordination needed |
| Offline batch queue | Collect locally, submit when connected |
| Sensor/IoT telemetry | Append-only, massive volume |

## 5. GC Workflows — When and How

### GC happens on natural sync points

GC is triggered on **every merge, pull, and push** — not on a timer, not as a background daemon. These operations already imply "the user has some seconds available and is not expecting an immediate response." They are the natural moments to ensure vault health.

```
push:   upload changes → GC (drain pending, rebuild indexes, run health check)
pull:   fetch remote → GC → merge
merge:  GC → integrate pending into branch tree → update ref
```

This gives eventual consistency without needing a scheduler or daemon. Between these operations the vault may have unprocessed change packs — that's fine. They're signed, they're safe, they'll be integrated on the next natural sync.

### GC responsibilities (in order)

```
1. COLLECT      — fetch all pending/{branch_id}/* objects
2. VERIFY       — check signature on each change pack (reject invalid ones)
3. ORDER        — sort by timestamp (best-effort, UTC, application handles strict ordering)
4. INTEGRATE    — build real commit(s) from change packs, link to branch tree
5. UPDATE REF   — advance refs/heads/{branch_id} to new commit
6. REBUILD INDEX — refresh branches.json, pending_counts.json, temporal indexes
7. HEALTH CHECK — sign a health check record (see Section 6 on transactions)
8. PRUNE        — delete processed pending objects
```

GC is **idempotent** — running it twice produces the same result because change pack keys are content-addressed. Only after a successful ref update + health check are pending objects pruned.

### No ordering guarantee from the vault

The vault does not guarantee ordering of change packs. UTC timestamps are used for best-effort ordering within GC, but clock skew across devices is real. Applications that need strict ordering include a monotonic sequence counter in their payload — that's application-layer business logic, not vault responsibility.

All timestamps inside the vault are **UTC**. Local timezone conversion is the application's problem.

## 5. Storage Layout — Unified Across All Contexts

This layout is **identical** whether the vault lives on the SG/Send server, locally as a bare clone, or packed as a zip. The same structure, the same object addressing, the same encryption. The only difference is the transport layer underneath.

```
server API:    PUT /api/vault/write/{vault_id}/{file_id}
local bare:    .sg_vault/{file_id}  (same file_id, same content)
zip:           vault.sg-vault.zip/{file_id}  (same file_id, same content)
```

### Layout

```
vault/{vault_id}/
  │
  ├── refs/
  │   ├── heads/
  │   │   ├── main                        → commit_id (encrypted ref)
  │   │   ├── {branch_id_a}               → commit_id (encrypted ref)
  │   │   └── {branch_id_b}               → commit_id (encrypted ref)
  │   └── remotes/
  │       └── origin/{branch_id}          → commit_id (last known remote state)
  │
  ├── objects/
  │   ├── commits/
  │   │   └── {ab}/{commit_id}            → encrypted Schema__Object_Commit
  │   ├── trees/
  │   │   └── {ab}/{tree_id}              → encrypted Schema__Object_Tree
  │   ├── blobs/
  │   │   └── {ab}/{blob_id}              → encrypted file content
  │   └── branches/
  │       └── {branch_id}/
  │           ├── meta                    → branch metadata (created_at, label, type)
  │           └── (no private key here — private keys never leave the device)
  │
  ├── keys/
  │   ├── registry.json                   → {key_id: {branch_id, type, created}} (encrypted)
  │   ├── public/
  │   │   ├── {branch_id}.pub             → branch public key (readable with vault read_key)
  │   │   └── {named_branch_id}.pub       → named branch public key
  │   └── private/
  │       └── {named_branch_id}.enc       → double-encrypted named branch private key
  │                                         (user branches never store private keys here)
  │
  ├── pending/
  │   └── {branch_id}/
  │       └── {iso_timestamp}_{random}    → headless commit / change pack (inline payload)
  │
  └── index/
      ├── branches.json                   → {branch_id: {head, last_seen, type}} (encrypted)
      ├── pending_counts.json             → {branch_id: count} (encrypted, cheap to poll)
      ├── last_gc.json                    → {branch_id: {timestamp, commit_count}} (encrypted)
      │
      ├── temporal/                       ← NEW: time-sliced activity indexes
      │   ├── last_hour.json              → {branch_id: [commit_ids since 1h ago]} (encrypted)
      │   ├── last_24h.json               → {branch_id: [commit_ids since 24h ago]} (encrypted)
      │   ├── last_week.json              → {branch_id: [commit_ids since 7d ago]} (encrypted)
      │   └── last_month.json             → {branch_id: [commit_ids since 30d ago]} (encrypted)
      │
      └── local/                          ← NEW: per-clone state (not pushed to server)
          ├── last_pull.json              → {branch_id: {commit_id, timestamp}}
          ├── last_push.json              → {branch_id: {commit_id, timestamp}}
          └── processed_refs.json         → {ref: last_processed_commit_id}
```

### Why the temporal indexes matter

Without them, answering "what changed in the last hour?" requires walking the full commit chain from HEAD backwards, decrypting each commit object, and filtering by timestamp. On a vault with thousands of commits that's expensive.

With `temporal/last_hour.json` it's one read — and because all IDs are content-addressed, the list of commit_ids in that file lets you fetch exactly the objects you need.

**Update strategy:** Temporal indexes are maintained by GC. When GC runs it recomputes the windows based on commit timestamps. Between GC runs they may be slightly stale — they're an optimisation, not ground truth. A client that needs exact precision walks the commit chain; a client that wants a quick "anything new since last hour?" uses the index.

**Useful scenarios:**
- Dashboard showing vault activity without full clone
- Agent deciding whether to pull before starting work
- Billing service computing activity windows
- "Show me what changed since my last session" on CLI open

### The `local/` sub-index

The `local/` directory lives only in the local clone (bare or working), never pushed to the server. It tracks the clone's own sync state:

- `processed_refs.json` — which remote refs this clone has already integrated (so pull knows what's new)
- `last_pull/push` — timestamps and commit_ids for display in `status` and `log`

This is the same information that git stores in `.git/FETCH_HEAD` and `.git/refs/remotes/origin/*`, but more explicit and structured.

## 6. Transactions and Vault Integrity

### The concurrency problem

The only real concurrency risk is when two clients try to update the **same ref** at the same time (e.g., two agents both pushing to `refs/heads/billing-branch` simultaneously). Everything else in the vault is immutable content-addressed objects — blobs, trees, commits all have unique IDs and can be written concurrently without conflict.

The critical observation: **almost every file in the vault has a unique content-addressed ID and is written once, never overwritten**. The only mutable files are:
- `refs/heads/{branch_id}` — the branch HEAD pointer
- `index/*.json` — the derived indexes

This is the same model as Git. And Git's answer is the same as ours: the client must `pull + merge` before `push` if the remote has moved forward. There is no concurrent write problem for blobs/trees/commits because they're immutable once written.

### Transaction model

```
Mutable (small set, require optimistic locking):
  refs/heads/{branch_id}          → branch HEAD
  index/branches.json             → branch registry
  index/pending_counts.json       → pending counts
  index/temporal/*.json           → temporal indexes

Immutable (large set, write-once, never overwritten):
  objects/commits/{id}            → commit objects
  objects/trees/{id}              → tree objects
  objects/blobs/{id}              → file blobs
  pending/{branch_id}/{ts}_{rnd}  → change packs (written once, deleted after GC)
  keys/public/{id}.pub            → public keys
```

For the mutable files, a simple **optimistic lock** suffices:
1. Read current value + its ETag/hash
2. Compute new value
3. Write with `If-Match: {etag}` (or equivalent) — server rejects if value changed since read
4. On rejection: retry (re-read, recompute, re-attempt)

For refs: this is exactly the git `push` rejection model. "Your push was rejected because the remote has diverged — pull and merge first."

### Health check after sync operations

After every GC run (which happens on push/pull/merge), a **signed health record** is written:

```json
{
  "schema": "vault_health_v1",
  "branch_id": "{branch_id}",
  "head_commit": "{commit_id}",
  "tree_hash": "{hash of current tree}",
  "pending_count": 0,
  "gc_completed_at": "2026-03-12T14:30:00Z",
  "signed_by": "{branch_key_fingerprint}",
  "signature": "..."
}
```

Stored at `index/health/{branch_id}_latest.json`. This lets any observer (or future GC run) verify the vault is in a known good state without re-reading all objects. It's also the signal that `pending_count: 0` means GC completed cleanly.

If a push fails partway through (crash after writing some blobs but before updating the ref), the vault is in a safe state: the partial blobs are orphaned immutable objects that GC will eventually prune. The ref was never updated, so the branch HEAD is still valid. No rollback needed — the immutable object model provides this for free.

## 7. File System Architecture (Sub-Tree Objects)

> **This topic has its own companion document** — see `v0.13.30__arch-exploration__vault-filesystem.md` for the full treatment of:
> - Why the current tree.json / Schema__Object_Tree duality must be unified
> - The sub-tree object model (git-style graph traversal replacing monolithic tree.json)
> - Path encryption: names are plaintext in tree.json today; the unified format encrypts them
> - How the "Thinking in Graphs" whitepaper applies — the vault as a graph of content-addressed nodes
> - Zip-based clone and server-side cache (how clone goes from 27 API calls to 1)
> - Migration path from current tree.json to unified sub-tree objects

**Summary of key decisions that flow into this document:**
- The unified format uses **sub-tree objects** (one tree object per directory level, content-addressed)
- **Path names are encrypted** in tree entries — the server sees only opaque IDs and encrypted names
- `_flatten_tree()` and `_build_tree_json()` are removed in the unified format (no translation layer)
- The `Schema__Object_Tree` flat list remains the in-memory working form; the graph is the storage form
- Full-tree enumeration is rare; the server-cached zip handles clone; common ops use partial traversal

## 8. Request Budget — Counting What Matters

For each workflow, we should track the API request count as a first-class metric. Target numbers:

| Workflow | Target requests | Notes |
|----------|----------------|-------|
| Headless commit (inline payload ≤256KB) | **1** | Single PUT to pending/ |
| Read one known file (cloneless) | **2** | tree + file |
| Read one file (file_id known) | **1** | Direct blob fetch |
| Push (existing branch, few changes) | **2-5** | changed blobs + tree + ref update |
| Clone (zip cached) | **1** | Single zip download |
| Clone (no cache, 25 files) | **27** | Current state (to fix) |
| GC (drain N pending commits) | **N+3** | fetch N, write tree, write commit, update ref |
| Check if pending exists | **1** | Read index/pending_counts.json |

The discipline: **every new workflow gets a request budget before implementation**.

## 9. Resolved and Open Questions

### ✅ Resolved: Change pack terminology

**Change pack** it is. Communicates a self-contained bundle of one or more changes. "Headless commit" dropped.

### ✅ Resolved: Lost branch key = permanently locked

No recovery, no retirement ceremony. A locked branch is a normal lifecycle state, not a failure. The workflow is:
- Branch key intentionally deleted → branch is permanently locked (read-only forever)
- Branch key accidentally lost → same outcome
- To continue writing: clone again → new branch, new key pair

There is no `branch retire` command. Locking is done by deleting the key. The locked state is self-evident from the absence of activity (no new commits) and optionally recorded in the index by any entity with vault access.

### ✅ Resolved: Named branch private key — common case

In the **common/simple case**, the named branch private key IS derived from the vault key. Anyone with the vault key can merge into `main`. Transparent, no extra ceremony.

In the **locked named branch case**: the private key is additionally wrapped with a specific user's public key. If that user's key is lost, the named branch is permanently locked (same as a user branch). Vault-level fallback (also encrypted with vault read_key) is the default unless explicitly opted out.

Multi-sig / quorum: noted as future design space. Not in scope for initial implementation.

### ✅ Resolved: Eventual consistency is fine — GC on sync operations

GC runs on every merge, pull, and push. These are natural moments where seconds are available. Temporal indexes may be stale between GC runs — that's acceptable. `pending_counts.json` provides a lightweight "are there unprocessed changes?" signal between GC runs.

### ✅ Resolved: No ordering guarantee from vault

Vault provides "all change packs, unordered." UTC timestamps used for best-effort GC ordering. Applications that need strict ordering embed a monotonic counter in the payload. All timestamps inside the vault are UTC — local timezone is the application's responsibility.

### ✅ Resolved: GC concurrency via git model, not leases

The immutable object model means **almost everything is safe to write concurrently** — blobs, trees, commits all have unique content-addressed IDs. The only mutable state is the handful of ref files and index files. For those, the standard git model applies: last writer wins on blobs, but ref updates are guarded by version check (reject if ref changed since you last read it). A failed push → pull first → retry. No GC lease needed for the common case.

### 🔲 Open: Path encryption confirmation

Tree.json today has plaintext file paths as keys. In the unified format we should encrypt them. The only thing visible to the server would then be opaque IDs and sizes — no filenames, no folder structure. This is a one-time format decision that cascades into all tree operations.

**Confirm: yes, encrypt paths in unified format?**

### 🔲 Open: Storage layout section numbering (low-stakes)

Sections above are currently 2-3-4-5-6-7. Need to reconcile numbering after this revision. Not architectural.
