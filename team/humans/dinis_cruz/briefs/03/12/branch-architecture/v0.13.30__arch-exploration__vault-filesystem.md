## 1. The Problem With the Current tree.json

The vault currently has two representations of the same thing that must be kept in sync:

### Representation 1: Server tree.json (nested, plaintext paths)

```json
{
  "version": 10,
  "updated": "2026-03-10T23:17:45.000Z",
  "tree": {
    "/": {
      "type": "folder",
      "children": {
        "README.md":  { "type": "file", "file_id": "1dc62e8eba09", "size": 153 },
        "experiments": {
          "type": "folder",
          "children": {
            "results": {
              "type": "folder",
              "children": {
                "run-001.txt": { "type": "file", "file_id": "0835ba70801e", "size": 216 }
              }
            }
          }
        }
      }
    }
  }
}
```

This is a **monolithic, nested, schema-first document**. The folder hierarchy is declared. File names are plaintext. The entire tree must be read to know anything about the vault.

### Representation 2: Schema__Object_Tree (flat, internal)

```python
class Schema__Object_Tree(Type_Safe):
    entries: list[Schema__Object_Tree_Entry]

class Schema__Object_Tree_Entry(Type_Safe):
    path    : Safe_Str__File_Path    # "experiments/results/run-001.txt"
    blob_id : Safe_Str__Object_Id    # content-addressed encrypted blob
    size    : Safe_UInt__File_Size
```

This is a **flat list of (path, blob_id, size) tuples**. All path information is embedded in the path string. No nesting, no folder structure — just an enumeration of every file.

### The translation tax

`Vault__Sync._flatten_tree()` exists only because these two formats don't agree. It recursively walks the nested server format to produce the flat internal format. This function is a symptom of the duality, not a feature.

```python
def _flatten_tree(self, tree: dict, prefix: str = '') -> dict:
    result = {}
    for name, node in tree.items():
        if name == '/':
            result.update(self._flatten_tree(node.get('children', {}), prefix))
        elif node.get('type') == 'folder':
            child_prefix = f'{prefix}{name}/' if prefix else f'{name}/'
            result.update(self._flatten_tree(node.get('children', {}), child_prefix))
        elif node.get('type') == 'file':
            path = f'{prefix}{name}'
            result[path] = node
    return result
```

Every code path that touches the tree — clone, push, pull, status, merge — pays this translation tax. The unification goal is to eliminate it entirely.

## 2. Thinking in Graphs: The Right Frame

The "Thinking in Graphs" whitepaper (Issues-FS, Feb 2026) articulates the core principle:

> "A node in a graph is just a node. What something 'is' emerges from the edges you can trace from it. Meaning is not declared — it is discovered through graph relationships."

Applied to the vault: a **folder is not a type declaration**. A folder is just a tree node that references other tree nodes instead of a blob. The folder "exists" only as an edge pattern in the graph. This is not a subtle distinction — it changes the entire storage and traversal model.

Compare:

**Schema-first (current tree.json):**
```
"experiments" has type "folder" and has children [...]
```
The folder declares itself. The structure is encoded as properties.

**Graph-first (proposed):**
```
tree_root → { "experiments" → tree_ab1234, "README.md" → blob_5f6789 }
tree_ab1234 → { "results" → tree_cd9012, "hypothesis.md" → blob_3e4567 }
tree_cd9012 → { "run-001.txt" → blob_0835ba }
```
The folder exists only as a tree node that points to other nodes. No type declaration. No "folder" concept baked into the schema. The structure IS the graph.

This is the exact model that git has used since 2005. It is not novel — it is proven at massive scale.

## 3. The Proposed Model: Sub-Tree Objects

### Core data types

```
blob    — encrypted file content, identified by content hash
tree    — encrypted list of { name → blob_id | tree_id } entries
commit  — encrypted { tree_id, parent_ids[], message, timestamp, branch_fingerprint, signature }
```

That's the entire model. Everything else — folders, file paths, directory hierarchy — is emergent from the graph structure.

### What a tree object contains

A tree object is an encrypted map of names to either blob IDs (files) or sub-tree IDs (folders):

```json
{
  "entries": [
    { "name_enc": "<encrypted 'README.md'>",    "blob_id": "5f6789..." },
    { "name_enc": "<encrypted 'experiments'>",  "tree_id": "ab1234..." },
    { "name_enc": "<encrypted 'notes'>",        "tree_id": "cd5678..." }
  ]
}
```

Names are encrypted (see Section 5 on path encryption). The distinction between "this is a file" and "this is a folder" is structural — entries have either `blob_id` (leaf) or `tree_id` (subtree).

### Graph traversal to reconstruct a path

```
Want: "experiments/results/run-001.txt"

1. Read root tree (from refs/heads/{branch} → commit → tree_id)
2. Find entry where decrypt(name_enc) == "experiments" → tree_id = ab1234
3. Read tree ab1234
4. Find entry where decrypt(name_enc) == "results" → tree_id = cd9012
5. Read tree cd9012
6. Find entry where decrypt(name_enc) == "run-001.txt" → blob_id = 0835ba
7. Read and decrypt blob 0835ba → file content
```

Cost: depth + 1 reads (one per folder level, plus the blob). For `experiments/results/run-001.txt` that's 4 reads. For a top-level file it's 2 reads (root tree + blob).

Compare to cloneless today: 3 API calls (settings + tree + file). With sub-trees: same 3 calls for a shallow file, 4-5 for a deep one. Acceptable.

## 4. Why This Is Better: The Five Benefits

### Benefit 1: Natural delta push

When you change one file in `experiments/results/run-001.txt`:

- A new blob is created (new content)
- A new tree is created for `results/` (points to new blob)
- A new tree is created for `experiments/` (points to new results tree)
- A new root tree is created (points to new experiments tree)
- A new commit is created (points to new root tree)
- All other blobs and sub-trees are **unchanged and shared**

You upload only the changed objects. Everything untouched is already on the server, already in the object store. The push is a delta by construction — no diff algorithm needed.

```
Before: root_tree_A → experiments_tree_A → results_tree_A → blob_A
After:  root_tree_B → experiments_tree_B → results_tree_B → blob_B
                   ↘ notes_tree (unchanged, shared)
                      README.md blob (unchanged, shared)
```

The unchanged sub-trees have the same content-addressed IDs in both commits. They don't need to be uploaded again.

### Benefit 2: Automatic deduplication across branches

Two branches that share most content share the same sub-tree objects:

```
Branch A (main):    root_A → { experiments → tree_X, notes → tree_Y, configs → tree_Z }
Branch B (feature): root_B → { experiments → tree_X, notes → tree_Y, configs → tree_W }
```

`tree_X` (experiments) and `tree_Y` (notes) are the same object on the server — one copy, referenced by both branches. Only the configs sub-tree differs. This is not an optimization that needs to be implemented — it falls out of content addressing automatically.

### Benefit 3: Granular access and lazy loading

A client that only needs `configs/EC2.json` traverses only the `configs/` subtree. It never reads the `experiments/` or `notes/` subtrees. For large vaults this is significant — a vault with 1000 files in `experiments/` and 3 files in `configs/` costs 2 API reads for the configs case, not 1003.

### Benefit 4: Cheap merge at folder boundaries

Three-way merge becomes a tree comparison at each node:

```
Base tree:   root_base → { experiments → tree_X_base, notes → tree_Y_base }
Branch A:    root_A    → { experiments → tree_X_A,    notes → tree_Y_base }
Branch B:    root_B    → { experiments → tree_X_base, notes → tree_Y_B    }
```

`experiments` sub-tree changed in A but not B → take A's version, no conflict.
`notes` sub-tree changed in B but not A → take B's version, no conflict.

Comparing `tree_id` values (content hashes) is sufficient to detect changes. If the tree_id is the same, the entire subtree is identical — no need to inspect it further.

### Benefit 5: The vault IS the graph (no translation layer)

The internal object store and the vault's logical structure are the same thing. There is no `_flatten_tree()`. There is no `_build_tree_json()`. There is no translation between "server format" and "internal format." A tree object in the object store IS the folder structure. Reading the vault IS traversing the graph.

This eliminates the entire format duality identified in the v0.13.30 architectural review.

## 5. Path Encryption

### What leaks today

In the current tree.json, file names and folder structure are plaintext:

```json
"experiments/results/run-001.txt"
```

The server — or anyone who intercepts tree.json — can read the complete file system structure. File contents are encrypted. Filenames are not. This is a metadata leak.

For many use cases this is acceptable. For some (medical records, legal documents, sensitive configs) the filenames themselves are sensitive: `patient_records/john_doe_2026.pdf` or `acquisition/target_company_analysis.pdf` reveals information before a single byte of content is decrypted.

### The fix: encrypt names in tree entries

In the unified format, names within tree entries are encrypted:

```json
{
  "entries": [
    {
      "name_enc": "AES-GCM({read_key}, 'experiments')",
      "tree_id":  "ab1234cd5678"
    },
    {
      "name_enc": "AES-GCM({read_key}, 'README.md')",
      "blob_id":  "5f6789ef0123",
      "size_enc": "AES-GCM({read_key}, 153)"
    }
  ]
}
```

The server sees: a list of opaque entries, each with an encrypted name and either a tree_id or blob_id. It knows:
- How many entries are in each tree (number of files/folders in a directory)
- The content-addressed IDs (which tells it nothing without the read key)
- The approximate file sizes (can be encrypted too if needed)

It does not know: any filenames, any folder names, any path structure.

### Traversal still works

Path traversal requires the read key:

```
Want "experiments/results/run-001.txt":

1. Read root tree, decrypt all name_enc values with read_key
2. Find entry where decrypted name == "experiments" → tree_id = ab1234
3. Read tree ab1234, decrypt name_enc values
4. Find "results" → tree_id = cd9012
5. Read tree cd9012, decrypt name_enc values
6. Find "run-001.txt" → blob_id = 0835ba
7. Read blob 0835ba, decrypt → file content
```

The decryption cost per tree level is trivial — AES-GCM on a short string. The API call count is unchanged.

### Index files: paths are IDs, not plaintext

Index files (like `index/branches.json`) reference tree entries by their content-addressed IDs, not by plaintext paths. The temporal indexes reference commit_ids. The only place a human-readable path appears is inside decrypted objects, after the read_key has been applied.

## 6. The Flat Schema__Object_Tree as Working Form

The sub-tree object model is the **storage format**. The flat `Schema__Object_Tree` (list of `{path, blob_id, size}`) remains the natural **in-memory working form** for operations that need the full file list:

- Status diff: compare flat list of (path, content_hash) pairs
- Push: determine which paths changed, which are new, which deleted
- Merge: three-way diff on flat path lists

The translation from graph to flat list is:

```python
def tree_to_flat(root_tree_id, object_store, read_key, prefix='') -> dict:
    tree = object_store.read_tree(root_tree_id, read_key)
    result = {}
    for entry in tree.entries:
        name = decrypt(entry.name_enc, read_key)
        if entry.blob_id:
            result[f'{prefix}{name}'] = {'blob_id': entry.blob_id, 'size': entry.size}
        elif entry.tree_id:
            result.update(tree_to_flat(entry.tree_id, object_store, read_key, f'{prefix}{name}/'))
    return result
```

This replaces `_flatten_tree()`. The difference: `_flatten_tree()` operates on the legacy nested JSON format. `tree_to_flat()` operates on real encrypted objects in the content-addressed store. Same output, fundamentally different inputs.

For the **common cases** (read one file, check if a folder changed), the flat list is never built — traversal stops at the relevant subtree.

## 7. How Often Do You Really Need the Full Tree?

This is the key question. The answer determines whether the per-level traversal cost is acceptable.

### Rare: Full tree enumeration

| Operation | Needs full tree? | Notes |
|-----------|-----------------|-------|
| `ls` (list all files) | Yes | But only for display, not for computation |
| Clone | Yes | But server generates zip once, cache invalidation per commit |
| Full status diff | Yes | Only if everything might have changed |
| Backup / export | Yes | Rare, high-latency-acceptable operation |

### Common: Partial traversal sufficient

| Operation | Needs full tree? | What's actually needed |
|-----------|-----------------|----------------------|
| Read one file | No | Path traversal to that file: depth + 1 reads |
| Push changed files | No | Only subtrees on the changed paths |
| Check if folder changed | No | Compare subtree ID: 1 comparison |
| Merge (no conflicts) | No | Compare subtree IDs at each level |
| `ls {folder}` | No | Read one subtree object |
| Temporal "what changed" | No | Read temporal index, then fetch those commit objects |
| Change pack submission | No | No tree needed at all |

The insight: **the cases that need the full tree are either rare or are handled by the server-cached zip**. The common, high-frequency operations (read a file, push a change, check a folder) need only partial traversal.

## 8. Storage Layout for Tree Objects

Tree objects live in the same content-addressed object store as blobs and commits:

```
vault/{vault_id}/
  └── objects/
      ├── commits/
      │   └── {ab}/{commit_id}        → encrypted Schema__Object_Commit
      ├── trees/
      │   └── {ab}/{tree_id}          → encrypted Schema__Object_Tree (sub-tree node)
      └── blobs/
          └── {ab}/{blob_id}          → encrypted file content
```

The `{ab}/` prefix is the first two hex chars of the ID — the same sharding pattern git uses to avoid filesystem inode limits.

### Object sizes

| Object type | Typical size | Notes |
|-------------|-------------|-------|
| Small blob | 100 bytes – 1 MB | Most config/document files |
| Large blob | 1 MB – 500 MB | Media, archives |
| Tree node | 50 – 2000 bytes | Each entry ~50 bytes encrypted |
| Commit | 200 – 500 bytes | Fixed schema, small |

A tree node with 20 entries is roughly 1 KB encrypted. A vault with 1000 files across 50 folders has ~50 tree objects totalling ~50 KB. The tree overhead is negligible.

## 9. Zip-Based Clone and the Server-Side Cache

The full-tree traversal cost is almost entirely absorbed by the **server-side zip cache**:

```
First clone of commit X:
  Server: traverse all objects reachable from commit X → generate zip → cache at index/zips/{commit_id}.zip
  Client: GET index/zips/{commit_id}.zip → 1 HTTP request → unpack locally

Second clone of same commit X:
  Client: GET index/zips/{commit_id}.zip → cache hit → 1 HTTP request

New commit Y (only 3 files changed from X):
  Server: compute delta between X and Y → update zip (not regenerate from scratch)
  Client: GET index/zips/{commit_id}.zip → 1 HTTP request
```

The zip is the full-tree materialisation. It is generated once, server-side, and cached. Every client that clones gets the same zip. The cost of "traversing the whole tree" is paid exactly once per commit, not once per clone.

This means:
- Clone latency goes from 27 API calls (current) to 1 API call (zip download)
- The sub-tree object model doesn't penalise clone performance at all
- The server can generate zips incrementally (diff from previous zip) rather than rebuilding from scratch

## 10. Graph Properties: What the Vault Inherits

By adopting the graph model, the vault acquires properties that would require significant engineering effort to add otherwise:

### Structural sharing (deduplication)

Objects with the same content have the same ID. A file that exists identically in 10 branches is stored once. A folder structure that is identical in main and a feature branch shares all sub-tree objects. No explicit deduplication logic needed.

### Immutability by construction

Objects are written once and never modified. A commit's tree_id is a cryptographic commitment to the exact state of all files at that point. Changing any file changes the blob_id, which changes the tree_id, which changes the commit_id. Tampering is detectable without a separate integrity check.

### History traversal

Walking backward through commit history is graph traversal: follow parent_id pointers. The entire history of a vault is reachable from the current HEAD. There is no separate "history" data structure — the graph IS the history.

### Subtree history

Because sub-trees are content-addressed, you can answer "when did the `configs/` folder last change?" by walking commit history and comparing the tree_id for the `configs/` entry. This is O(commits) comparisons, no full-tree reads.

### Confidence through connectivity

The whitepaper's core insight: confidence is proportional to connectivity. A vault file's meaning is not just in its content — it is in:
- Which commit it appears in (and that commit's signature)
- Which branch that commit is on (and that branch's key fingerprint)
- Which merge brought it into main (and when)
- Whether its content_hash matches the stored hash (integrity)

Each of these is an edge in the graph. More edges = more confident understanding of what the file is, where it came from, and whether it has been tampered with.

## 11. Migration Path from Current tree.json

The current tree.json is the server-side source of truth. Migration to sub-tree objects is the core of the v0.13.30 "bare vault unification" brief.

### Phase 1: Read both, write tree objects only

- During clone: read tree.json to initialise object store with tree objects
- During push: write tree objects to object store, also write legacy tree.json (for backward compat with server UI)
- Server UI continues to use tree.json for display

### Phase 2: Generate tree.json from tree objects

- Server generates tree.json on the fly from root tree object (or on push)
- tree.json becomes derived data, not primary storage
- A `tree_to_nested_json()` function produces tree.json from the graph (reverse of `_flatten_tree`)

### Phase 3: Remove legacy tree.json dependency

- Server UI updated to read from object store directly (or from generated tree.json)
- `_flatten_tree()` removed
- `_build_tree_json()` removed
- Vault__Legacy_Guard.py scope reduced

This is a three-phase migration, not a flag day. Each phase is independently verifiable.

## 12. Open Questions

### Q1: Sub-tree granularity — one tree per folder, or configurable?

One tree object per directory level (the git model) is clean and consistent. An alternative is to allow "wide" tree objects that represent multiple levels — useful for very shallow vaults (1-2 levels deep) where the per-level object overhead is unnecessary.

**Recommendation:** Start with git model (one tree per folder level). Optimise only if benchmarks show overhead at shallow depths.

### Q2: Size metadata — in tree entry or only in blob?

Currently `size` is in the tree entry (so you can show file sizes in `ls` without reading blobs). With encrypted names, size is also encrypted — so listing files still requires decrypting tree entries, just not fetching blobs.

Should `size` stay in the tree entry (as an encrypted field), or move to a separate index?

**Recommendation:** Keep encrypted size in tree entry. It's small, and `ls` being a pure local operation (no extra reads) is valuable.

### Q3: content_hash in tree entry?

The v0.13.20 brief proposed adding `content_hash = SHA256(plaintext)[:12]` to `Schema__Object_Tree_Entry` for change detection (replaces the current size-only comparison). With path encryption, this field would also be encrypted.

**This is a strong yes** — it fixes the silent miss of same-size edits and makes `status` reliable. Small additional storage cost per entry.

### Q4: Tree object format — JSON or binary?

Currently all vault objects are JSON encrypted with AES-256-GCM. For tree objects this is fine — they're small. Binary encoding (msgpack, protobuf) would be more compact but adds parsing complexity.

**Recommendation:** Stay with JSON. Compactness of tree objects is not a bottleneck. Keep the format consistent.

## 13. Relationship to Branch Model

See companion document `v0.13.30__arch-exploration__branch-model.md` for how tree objects connect to commits, branches, and change packs.

The key connection points:
- A **commit** references a **root tree** (its tree_id)
- A **change pack** may carry inline blob data that GC integrates into tree objects on drain
- A **named branch** merge creates a new root tree that references the best sub-tree objects from both parent commits
- **Temporal indexes** reference commit_ids, which reference tree_ids — traversal from index to tree to file is always possible
