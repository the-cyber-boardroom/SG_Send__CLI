## 0. The Three Primitives

Everything the vault does maps to three operations on opaque encrypted blobs.
The server has no knowledge of what it stores — no filenames, no structure, no content.

```
┌──────────────────────────────────────────────────────────────┐
│                     api/vault                                │
│                                                              │
│  PUT    /api/vault/write/{vault_id}/{file_id}    ← AUTH      │
│         Body: encrypted blob                                 │
│         Header: x-sgraph-vault-write-key                     │
│                                                              │
│  GET    /api/vault/read/{vault_id}/{file_id}     ← NO AUTH   │
│         Returns: encrypted blob (raw binary)                 │
│                                                              │
│  DELETE /api/vault/delete/{vault_id}/{file_id}   ← AUTH      │
│         Removes: encrypted blob                              │
│                                                              │
│  (also: GET /api/vault/read-base64/...  for MCP/AI compat)   │
└──────────────────────────────────────────────────────────────┘

  vault_id  = public identifier (from vault key, after the colon)
  file_id   = content-addressed hash OR derived deterministic ID
  write_key = HMAC derived from vault passphrase (never the passphrase itself)
  read      = unauthenticated (blob is encrypted — possessing file_id is harmless)
```

**New endpoints needed for branch model:**

```
  GET    /api/vault/list/{vault_id}/{prefix}       ← NO AUTH
         Returns: list of file_ids matching prefix
         Why: enumerate refs/, keys/public/, pending/{branch_id}/
         (without this, clients must know exact file_ids)

  PUT    /api/vault/write-if-match/{vault_id}/{file_id}   ← AUTH
         Header: If-Match: {etag}
         Why: optimistic locking for mutable refs (branch HEADs)
         Rejects write if blob changed since last read
```

## 1. The Unified Storage Layout

**The core invariant:** the layout is identical whether stored on the SG/Send
server, on local disk as a bare vault, or packed as a portable zip.
The only difference is the transport underneath.

```
                    ┌───────────────────────────┐
                    │    SAME LAYOUT EVERYWHERE │
                    └──────────┬────────────────┘
                               │
           ┌───────────────────┼────────────────────┐
           │                   │                    │
           ▼                   ▼                    ▼
    ┌─────────────┐    ┌──────────────┐    ┌──────────────┐
    │  SG/Send    │    │  Local Disk  │    │   Zip File   │
    │  (Cloud)    │    │  (Bare)      │    │  (Portable)  │
    ├─────────────┤    ├──────────────┤    ├──────────────┤
    │ PUT write/  │    │ write to     │    │ add entry    │
    │   {id}/{fid}│    │ .sg_vault/   │    │ to archive   │
    │ GET read/   │    │   {fid}      │    │   {fid}      │
    │   {id}/{fid}│    │ read from    │    │ read entry   │
    │ DELETE del/ │    │ .sg_vault/   │    │ from archive │
    │   {id}/{fid}│    │   {fid}      │    │   {fid}      │
    └─────────────┘    └──────────────┘    └──────────────┘

    Backend abstraction: read(fid), write(fid, data), delete(fid), list(prefix)
```

### The layout itself

```
vault/{vault_id}/
│
├── refs/
│   ├── heads/
│   │   ├── main                           → commit_id (named branch, merge-only)
│   │   ├── {branch_id_a}                  → commit_id (user branch from clone)
│   │   └── {branch_id_b}                  → commit_id (agent branch from clone)
│   └── remotes/
│       └── origin/
│           └── main                       → commit_id (last known remote state)
│
├── objects/
│   ├── commits/
│   │   └── {ab}/{commit_id}               → encrypted Schema__Object_Commit
│   ├── trees/
│   │   └── {ab}/{tree_id}                 → encrypted Schema__Object_Tree (sub-tree)
│   └── blobs/
│       └── {ab}/{blob_id}                 → encrypted file content
│
├── keys/
│   ├── registry.json                      → {key_id: {branch_id, type, created}}  [encrypted]
│   ├── public/
│   │   ├── {branch_id}.pub               → branch EC P-256 public key  [encrypted w/ read_key]
│   │   └── main.pub                       → named branch public key
│   └── private/
│       └── main.enc                       → named branch private key  [double-encrypted]
│                                            outer: vault read_key
│                                            inner: (optional) user's public key
│
├── pending/
│   └── {branch_id}/
│       └── {iso_ts}_{random}              → change pack (signed, self-contained)
│
└── index/
    ├── branches.json                      → {branch_id: {head, last_seen, type}}  [encrypted]
    ├── pending_counts.json                → {branch_id: count}  [encrypted, cheap poll]
    ├── health/{branch_id}_latest.json     → signed health record
    ├── temporal/
    │   ├── last_hour.json                 → recent commit_ids by branch
    │   └── last_24h.json
    └── local/                             ← never pushed to server
        ├── last_pull.json
        ├── last_push.json
        └── processed_refs.json
```

## 2. The Object Model (Three Types)

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        OBJECT TYPES                                     │
│                                                                         │
│  blob      encrypted file content, ID = SHA256(ciphertext)              │
│  tree      encrypted list of { name_enc, blob_id|tree_id, size_enc }    │
│  commit    encrypted { tree_id, parents[], message, timestamp,          │
│                        branch_fingerprint, signature }                  │
│                                                                         │
│  Everything else — folders, paths, history — is emergent from           │
│  the graph of these three types.                                        │
└─────────────────────────────────────────────────────────────────────────┘


Example: vault with  README.md, configs/EC2.json, configs/deploy.yaml

                     ┌───────────────────────────┐
                     │  commit  c1a2b3           │
                     │  tree_id: root_tree       │
                     │  parents: [prev_commit]   │
                     │  signed_by: branch_fp     │
                     └────────────┬──────────────┘
                                  │
                     ┌────────────▼───────────────┐
                     │  tree (root)  t_root       │
                     │                            │
                     │  enc("README.md")  → blob  │
                     │  enc("configs")   → tree   │
                     └───────┬──────────┬─────────┘
                             │          │
                ┌────────────▼──┐  ┌────▼───────────────────┐
                │  blob b_read  │  │  tree (configs) t_cfg  │
                │  encrypted    │  │                        │
                │  README.md    │  │  enc("EC2.json") → blob│
                │  content      │  │  enc("deploy.yaml")→ b │
                └───────────────┘  └──────┬──────────┬──────┘
                                          │          │
                                  ┌───────▼───┐ ┌────▼───────┐
                                  │ blob b_ec2│ │ blob b_dep │
                                  │ encrypted │ │ encrypted  │
                                  │ EC2.json  │ │ deploy.yaml│
                                  └───────────┘ └────────────┘

  Server sees: opaque blobs at content-addressed IDs
  Server knows: nothing about names, paths, or content
  Client with read_key: can traverse the graph and decrypt everything
```

## 3. Key Derivation — One Secret Derives Everything

```
  Vault key:   xk8mp2vn9qrstu4567890abc:a1b2c3d4
               └──── passphrase (24) ───┘ └ vault_id ┘


  passphrase + vault_id
         │
         ▼  HKDF-SHA256
    ┌────┴──────────────────────────────────────────┐
    │                                               │
    ▼                    ▼                ▼          ▼
  read_key          write_key      tree_file_id   settings_file_id
  (AES-256          (HMAC for      (deterministic  (deterministic
   decryption)       API auth)      server ID)      server ID)

  The server receives:  vault_id (public) + write_key (for auth)
  The server never sees: passphrase, read_key
  The passphrase is the SINGLE SECRET for the entire vault
```

## 4. Flow: Init

Creates a new vault. Identical outcome on any backend.

```
  sg-send-cli init my-vault --token <token>

  ┌─────────┐
  │  User   │
  └────┬────┘
       │
       │  1. generate vault_key (passphrase:vault_id)
       │  2. derive read_key, write_key, file_ids from vault_key
       │  3. generate EC P-256 key pair for "main" named branch
       │  4. generate EC P-256 key pair for this clone's branch
       │
       ▼
  ┌───────────────────────────────────────────────────────────┐
  │  LOCAL: create .sg_vault/ structure                       │
  │                                                           │
  │  objects/trees/{root_tree_id}   ← empty tree (enc)        │
  │  objects/commits/{commit_id}    ← init commit (enc)       │
  │  refs/heads/main                ← commit_id               │
  │  refs/heads/{my_branch_id}     ← commit_id                │
  │  keys/public/main.pub          ← main public key          │
  │  keys/public/{my_branch}.pub   ← my public key            │
  │  keys/private/main.enc         ← main private key         │
  │                                   (encrypted w/ read_key) │
  │  VAULT-KEY                      ← vault key (local only)  │
  └──────────────────────┬────────────────────────────────────┘
                         │
                         │  5. push initial objects to remote
                         ▼
                ┌─────────────────────┐
                │  REMOTE (SG/Send)   │
                │                     │
                │  PUT write/{v}/{f}  │  × N objects
                │  (encrypted blobs)  │
                └─────────────────────┘

  Output:
    Initialized vault in my-vault/
    Vault key: xk8mp2vn9qrstu4567890abc:a1b2c3d4
    Branch:    {my_branch_id} (your clone's identity)

  Request budget: 1 (per object) + 1 (ref) = ~6-8 PUTs for init
```

## 5. Flow: Clone (Branch Auto-Creation)

Every clone creates a new branch with its own key pair.
The user never thinks about branches for basic workflows.

```
  sg-send-cli clone xk8mp2vn9qrstu4567890abc:a1b2c3d4

  ┌─────────┐                                ┌──────────────┐
  │  User   │                                │  SG/Send     │
  └────┬────┘                                └──────┬───────┘
       │                                            │
       │  1. derive keys from vault_key             │
       │  2. generate EC P-256 key pair             │
       │     for this clone's branch                │
       │                                            │
       │  3. GET read/{vault_id}/refs/heads/main    │
       │  ─────────────────────────────────────►    │
       │  ◄─────────────────────────────────────    │
       │     commit_id for main HEAD                │
       │                                            │
       │  4. GET zip (if cached) ─ OR ─             │
       │     walk commit → tree → blobs             │
       │  ─────────────────────────────────────►    │
       │  ◄─────────────────────────────────────    │
       │     all objects for this commit            │
       │                                            │
       │  5. PUT write keys/public/{new_branch}.pub │
       │  ─────────────────────────────────────►    │
       │     register my branch public key          │
       │                                            │
       │  6. PUT write refs/heads/{new_branch}      │
       │  ─────────────────────────────────────►    │
       │     point my branch at same commit         │
       │                                            │

  Local result:

    my-vault/
    ├── .sg_vault/
    │   ├── objects/           ← all objects from remote
    │   ├── refs/
    │   │   ├── heads/
    │   │   │   ├── main                 ← remote main HEAD
    │   │   │   └── {my_branch_id}      ← same commit (just cloned)
    │   │   └── remotes/origin/main     ← tracking ref
    │   ├── keys/public/{my_branch}.pub
    │   ├── VAULT-KEY
    │   └── branch_key.pem              ← my branch private key (NEVER pushed)
    ├── README.md              ← decrypted working copy
    └── configs/
        ├── EC2.json
        └── deploy.yaml

  Request budget:
    With zip cache:   1 GET (zip) + 2 PUTs (public key + ref)  = 3
    Without cache:    N GETs (objects) + 2 PUTs                 = N+2
    Current (v0.5.x): 27 GETs for 25 files                     = 27
```

## 6. Flow: Commit (New — Explicit)

Separates "snapshot my work" from "upload to server."
No staging area — commit always snapshots the full working directory.

```
  sg-send-cli commit -m "updated EC2 config"

  ┌──────────────────────────────────────────────────────┐
  │                   LOCAL ONLY                         │
  │                   (no network)                       │
  │                                                      │
  │  Working Dir              Object Store               │
  │  ┌──────────┐            ┌──────────────────┐        │
  │  │ README.md│──────────► │ blob (unchanged) │        │
  │  │ configs/ │            │ tree (unchanged) │        │
  │  │  EC2.json│──► new ──► │ NEW blob b_ec2v2 │        │
  │  │  deploy. │──────────► │ blob (unchanged) │        │
  │  └──────────┘            └────────┬─────────┘        │
  │                                   │                  │
  │  1. scan working dir              │                  │
  │  2. hash each file (SHA256)       │                  │
  │  3. compare content_hash to       │                  │
  │     committed tree entries        │                  │
  │  4. encrypt + store changed       │                  │
  │     blobs in object store         ▼                  │
  │                          ┌───────────────────┐       │
  │  5. build new sub-trees  │ NEW tree t_cfg2   │       │
  │     for changed paths    │ (configs/ changed)│       │
  │                          └────────┬──────────┘       │
  │                                   │                  │
  │                          ┌────────▼─────────┐        │
  │  6. build new root tree  │ NEW tree t_root2 │        │
  │                          └────────┬─────────┘        │
  │                                   │                  │
  │  7. create commit object ┌────────▼──────────┐       │
  │     sign with branch key │ NEW commit c_new  │       │
  │                          │ tree: t_root2     │       │
  │                          │ parent: [c_prev]  │       │
  │                          │ signed: branch_fp │       │
  │                          └────────┬──────────┘       │
  │                                   │                  │
  │  8. update local HEAD    refs/heads/{my_branch}      │
  │                          → c_new                     │
  └──────────────────────────────────────────────────────┘

  Request budget: 0  (entirely local)
```

## 7. Flow: Push (With Guards)

Upload committed changes to remote. Requires clean working directory
and up-to-date with remote (no diverged history).

```
  sg-send-cli push

  ┌─────────┐                               ┌──────────────┐
  │  Client  │                              │  SG/Send     │
  └────┬────┘                               └──────┬───────┘
       │                                           │
       │  1. CHECK: working dir clean?             │
       │     (no uncommitted changes)              │
       │     ✗ → "commit first"                    │
       │                                           │
       │  2. FETCH: read remote ref                │
       │  GET read/{v}/refs/heads/{my_branch}      │
       │  ─────────────────────────────────────►   │
       │  ◄─────────────────────────────────────   │
       │                                           │
       │  3. CHECK: local HEAD descends from       │
       │     remote HEAD?                          │
       │     ✗ → "pull first"                      │
       │                                           │
       │  4. DIFF: find objects in local that      │
       │     remote doesn't have                   │
       │     (new blobs, new trees, new commit)    │
       │                                           │
       │  5. UPLOAD: only the delta                │
       │  PUT write/{v}/objects/blobs/{new_blob}   │
       │  ─────────────────────────────────────►   │
       │  PUT write/{v}/objects/trees/{new_tree}   │
       │  ─────────────────────────────────────►   │
       │  PUT write/{v}/objects/commits/{new_cmt}  │
       │  ─────────────────────────────────────►   │
       │                                           │
       │  6. UPDATE REF (optimistic lock)          │
       │  PUT write-if-match/{v}/refs/heads/{br}   │
       │  ─────────────────────────────────────►   │
       │     If-Match: {old_commit_id}             │
       │     Body: {new_commit_id}                 │
       │                                           │
       │  7. GC: drain any pending change packs    │
       │     for this branch (see Flow 11)         │
       │                                           │
       │  8. UPDATE tracking ref                   │
       │     refs/remotes/origin/{my_branch}       │
       │     → new_commit_id                       │
       │                                           │

  Request budget (3 changed files, 2 tree levels):
    1 GET (remote ref check)
    + 3 PUTs (blobs) + 2 PUTs (trees) + 1 PUT (commit)
    + 1 PUT (ref update)
    = 8 requests

  Push to YOUR branch always succeeds (you're the only writer).
  Push to a NAMED branch (main) requires merge, not direct push.
```

## 8. Flow: Pull = Fetch + Merge

Two-phase operation. Fetch downloads remote state without touching
working directory. Merge integrates it.

```
  sg-send-cli pull

  ┌─ PHASE 1: FETCH ────────────────────────────────────────────────┐
  │                                                                 │
  │  ┌──────────┐                              ┌──────────────┐     │
  │  │  Client  │                              │  SG/Send     │     │
  │  └────┬─────┘                              └──────┬───────┘     │
  │       │                                           │             │
  │       │  GET read/{v}/refs/heads/{my_branch}      │             │
  │       │  ─────────────────────────────────────►   │             │
  │       │  ◄───── remote_commit_id ─────────────    │             │
  │       │                                           │             │
  │       │  (if remote == local → "already up to date")            │
  │       │                                           │             │
  │       │  GET objects we don't have locally        │             │
  │       │  (walk remote commit → tree → blobs)      │             │
  │       │  ─────────────────────────────────────►   │             │
  │       │  ◄─────────────────────────────────────   │             │
  │       │                                           │             │
  │       │  Store as FETCH_HEAD + update             │             │
  │       │  refs/remotes/origin/{my_branch}          │             │
  │       │                                           │             │
  │       │  Working directory: UNTOUCHED             │             │
  │       │                                           │             │
  └───────┴───────────────────────────────────────────┘

  ┌─ PHASE 2: MERGE (local only) ─────────────────────────────────┐
  │                                                               │
  │  Find common ancestor (LCA):                                  │
  │    Walk both commit chains backward until shared commit_id    │
  │    (fast path: use refs/remotes/origin/{br} before fetch)     │
  │                                                               │
  │           base (LCA)                                          │
  │          ╱            ╲                                       │
  │    local HEAD      FETCH_HEAD                                 │
  │          ╲            ╱                                       │
  │         merged result                                         │
  │                                                               │
  │  Three-way merge at sub-tree level:                           │
  │                                                               │
  │  For each path in union(base, local, remote):                 │
  │                                                               │
  │  ┌──────────┬─────────────┬───────────────────────────┐       │
  │  │ Local    │ Remote      │ Result                    │       │
  │  ├──────────┼─────────────┼───────────────────────────┤       │
  │  │ same     │ same        │ keep (no change)          │       │
  │  │ same     │ CHANGED     │ accept remote             │       │
  │  │ CHANGED  │ same        │ keep local                │       │
  │  │ ADDED    │ absent      │ keep local add            │       │
  │  │ absent   │ ADDED       │ accept remote add         │       │
  │  │ DELETED  │ same        │ keep delete               │       │
  │  │ same     │ DELETED     │ accept delete             │       │
  │  │ CHANGED  │ CHANGED     │ ⚡ CONFLICT                │       │
  │  │ DELETED  │ CHANGED     │ ⚡ CONFLICT                │       │
  │  │ CHANGED  │ DELETED     │ ⚡ CONFLICT                │       │
  │  └──────────┴─────────────┴───────────────────────────┘       │
  │                                                               │
  │  Conflict resolution:                                         │
  │    1. rename local → filename.conflict.ext                    │
  │    2. accept remote as filename.ext                           │
  │    3. report conflict to user                                 │
  │    4. merge always completes (never blocks)                   │
  │                                                               │
  │  Sub-tree shortcut:                                           │
  │    if tree_id(base) == tree_id(local) → whole folder          │
  │    unchanged locally → take remote version without            │
  │    inspecting individual files                                │
  │                                                               │
  │  Create merge commit:                                         │
  │    parents: [local_HEAD, FETCH_HEAD]   ← two parents          │
  │    tree_id: merged root tree                                  │
  │    signed_by: my branch key                                   │
  │                                                               │
  │  Update refs/heads/{my_branch} → merge_commit_id              │
  │  Update working directory with merged files                   │
  └───────────────────────────────────────────────────────────────┘

  Request budget:
    Fetch: 1 GET (ref) + N GETs (new objects) = N+1
    Merge: 0 (local only)
```

## 9. Flow: Branch Per Clone — The Identity Model

```
  ┌──────────────────────────────────────────────────────────────────┐
  │                    VAULT BRANCH TOPOLOGY                         │
  │                                                                  │
  │  Device 1 (laptop)           Device 2 (Lambda)                   │
  │  ┌───────────────┐          ┌──────────────────┐                 │
  │  │ clone → br_A  │          │ clone → br_B     │                 │
  │  │ key pair: kpA │          │ key pair: kpB    │                 │
  │  │ private: LOCAL│          │ private: LOCAL   │                 │
  │  └───────┬───────┘          └────────┬─────────┘                 │
  │          │                           │                           │
  │          │ commits signed by kpA     │ commits signed by kpB     │
  │          ▼                           ▼                           │
  │     ┌─ br_A ─┐                 ┌─ br_B ─┐                        │
  │     │ c1     │                 │ c1     │                        │
  │     │ c2     │                 │ c2     │                        │
  │     │ c3     │                 │ c3     │                        │
  │     └───┬────┘                 └───┬────┘                        │
  │         │                          │                             │
  │         │     merge (signed        │                             │
  │         │      by main key)        │                             │
  │         └──────────┐  ┌────────────┘                             │
  │                    ▼  ▼                                          │
  │              ┌─── main ───┐                                      │
  │              │ merge_cm_1 │ ← parents: [br_A/c3, br_B/c3]        │
  │              │ merge_cm_2 │                                      │
  │              └────────────┘                                      │
  │              merge-only branch                                   │
  │              private key in vault                                │
  │              (double-encrypted)                                  │
  └──────────────────────────────────────────────────────────────────┘


  Branch key lifecycle:

  clone()                    work                   done
    │                         │                       │
    ▼                         ▼                       ▼
  generate key pair    sign commits with it     delete private key
  push public key      push to own branch       branch = LOCKED
  to vault             (always succeeds)        (read-only forever)
                                                clone again if needed
```

## 10. Flow: Change Packs (The Performance Primitive)

For high-frequency writes that can't afford the full commit ceremony.
One HTTP request. No clone needed. No read access needed.

```
  ┌──────────────────────────────────────────────────────────────────┐
  │                    CHANGE PACK FLOW                              │
  │                                                                  │
  │  Normal commit:  read HEAD → diff → write blobs → write tree     │
  │                  → write commit → update ref   = 5+ requests     │
  │                                                                  │
  │  Change pack:    POST one signed blob          = 1 request       │
  └──────────────────────────────────────────────────────────────────┘


  ┌─ Creator (e.g. Lambda, webhook, CI)  ─┐     ┌─────────────────┐
  │                                       │     │    SG/Send      │
  │  Needs:                               │     │                 │
  │   • branch public key                 │     │                 │
  │   • own signing key                   │     │                 │
  │   • change pack schema                │     │                 │
  │                                       │     │                 │
  │  Does NOT need:                       │     │                 │
  │   • vault key                         │     │                 │
  │   • read access                       │     │                 │
  │   • clone                             │     │                 │
  │                                       │     │                 │
  │  1. build payload (JSON or zip)       │     │                 │
  │  2. encrypt with branch public key    │     │                 │
  │  3. sign with own key                 │     │                 │
  │  4. POST to pending/                  │     │                 │
  │                                       │     │                 │
  │  PUT write/{v}/pending/{br}/{ts}_{rnd}│────►│  stored as      │
  │                                       │     │  opaque blob    │
  └───────────────────────────────────────┘     └────────┬────────┘
                                                         │
                                                         │  (sits in pending/
                                                         │   until GC runs)
                                                         ▼
  ┌─ GC (runs on next push/pull/merge by any vault participant) ──────┐
  │                                                                   │
  │  1. COLLECT:  list pending/{branch_id}/*                          │
  │  2. VERIFY:   check signature on each change pack                 │
  │  3. ORDER:    sort by timestamp (best-effort UTC)                 │
  │  4. INTEGRATE: build real commit(s), link to branch tree          │
  │  5. UPDATE REF: advance refs/heads/{branch_id}                    │
  │  6. REBUILD INDEXES: branches.json, pending_counts.json           │
  │  7. HEALTH CHECK: sign health record                              │
  │  8. PRUNE: delete processed pending objects                       │
  └───────────────────────────────────────────────────────────────────┘

  The "keyless creator" capability:
    An external system that knows only the branch public key + its own
    signing key can write to the vault. No clone, no read, no vault key.
    The vault is a WRITE-ONLY APPEND TARGET for external systems.
```

## 11. Flow: Merge Into Named Branch (main)

Named branches are merge-only. Direct commits are cryptographically invalid.

```
  sg-send-cli merge main       (merge my branch into main)

  ┌─────────────────────────────────────────────────────────────────┐
  │                                                                 │
  │  1. READ: fetch main's HEAD and my branch's HEAD                │
  │                                                                 │
  │  2. DECRYPT: named branch private key (main.enc)                │
  │     outer layer: vault read_key                                 │
  │     inner layer: (if locked) user's private key                 │
  │                                                                 │
  │  3. FIND LCA: walk both parent chains to common ancestor        │
  │                                                                 │
  │  4. THREE-WAY MERGE at sub-tree level:                          │
  │                                                                 │
  │     base tree (LCA)                                             │
  │         │                                                       │
  │         ├── configs/  tree_id: t_cfg_base                       │
  │         │                                                       │
  │         ├── main's tree:    configs/ → t_cfg_base  (unchanged)  │
  │         └── my tree:        configs/ → t_cfg_new   (changed)    │
  │                                                                 │
  │         Result: take my version (main didn't change configs/)   │
  │         Shortcut: compare tree_ids, don't read individual files │
  │                                                                 │
  │  5. BUILD merge commit:                                         │
  │     tree_id: merged_root_tree                                   │
  │     parents: [main_HEAD, my_branch_HEAD]                        │
  │     signed_by: main's key pair  ← THIS IS THE PROTECTION        │
  │                                                                 │
  │     Only entities that can decrypt main.enc can sign this.      │
  │     A commit targeting main without main's signature            │
  │     is structurally invalid — not policy, cryptography.         │
  │                                                                 │
  │  6. UPDATE refs/heads/main → merge_commit_id                    │
  │     (optimistic lock: reject if main moved since we read it)    │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘


  Access control levels:

  ┌────────────────────────────────────────────────────────────────┐
  │  OPEN main:   private key encrypted with vault read_key only   │
  │               anyone with vault key can merge                  │
  │                                                                │
  │  LOCKED main: private key also encrypted with user's pub key   │
  │               only that user can decrypt → only they merge     │
  │                                                                │
  │  QUORUM:      key split across N holders (future)              │
  └────────────────────────────────────────────────────────────────┘
```

## 12. Flow: Portable Vault (Zip Distribution)

The vault travels as a file. The key travels as a message.
No server needed.

```
  ┌─────────────────────────────────────────────────────────────┐
  │                                                             │
  │  EXPORT                         IMPORT                      │
  │                                                             │
  │  sg-send-cli export             sg-send-cli clone           │
  │    my-vault/ → vault.sg.zip       vault.sg.zip              │
  │                                    --vault-key <key>        │
  │  ┌──────────────┐                                           │
  │  │ vault.sg.zip │     email / WhatsApp / Signal / AirDrop   │
  │  │              │  ──────────────────────────────────────►  │
  │  │ refs/...     │                                           │
  │  │ objects/...  │     "here's the vault key: xk8mp..."      │
  │  │ keys/...     │  ──────────────────────────────────────►  │
  │  │ index/...    │     (out-of-band, different channel)      │
  │  └──────────────┘                                           │
  │                                                             │
  │  Same layout inside the zip as on disk and on server.       │
  │  The recipient clones from the zip (local backend),         │
  │  then optionally adds a remote:                             │
  │                                                             │
  │  sg-send-cli remote add origin https://send.sgraph.ai       │
  │  sg-send-cli push origin                                    │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘
```

## 13. Flow: Remotes (Multi-Backend)

```
  sg-send-cli remote add origin https://send.sgraph.ai
  sg-send-cli remote add backup /mnt/external/vaults
  sg-send-cli remote add s3 s3://my-bucket/vaults

  ┌───────────────────────────────────────────────────────────────┐
  │  Backend Abstraction Layer                                    │
  │                                                               │
  │  interface:  read(file_id) → bytes                            │
  │              write(file_id, data) → ok                        │
  │              delete(file_id) → ok                             │
  │              list(prefix) → [file_ids]                        │
  │                                                               │
  │  ┌────────────────┐  ┌───────────────┐  ┌────────────────┐    │
  │  │ Backend__      │  │ Backend__     │  │ Backend__      │    │
  │  │ SG_Send        │  │ Local_Folder  │  │ Zip            │    │
  │  │                │  │               │  │                │    │
  │  │ PUT/GET/DELETE │  │ open/read/    │  │ zipfile.read/  │    │
  │  │ via HTTP       │  │ write to fs   │  │ write to zip   │    │
  │  │                │  │               │  │                │    │
  │  │ adapts:        │  │ file_id maps  │  │ file_id maps   │    │
  │  │ vault_id +     │  │ directly to   │  │ to zip entry   │    │
  │  │ file_id → URL  │  │ file path     │  │ path           │    │
  │  └────────────────┘  └───────────────┘  └────────────────┘    │
  │                                                               │
  │  Future: Backend__S3, Backend__URL (read-only)                │
  └───────────────────────────────────────────────────────────────┘

  Push/pull become remote-aware:

    sg-send-cli push origin        ← push to SG/Send
    sg-send-cli push backup        ← push to local folder
    sg-send-cli pull origin        ← pull from SG/Send
    sg-send-cli clone /path/to/vault.sg.zip   ← clone from zip
```

## 15. Request Budgets — Summary

| Workflow | Requests | Notes |
|----------|----------|-------|
| **init** | ~6-8 PUTs | objects + refs + keys |
| **clone (zip cached)** | 1 GET + 2 PUTs | zip + register branch |

| **push (3 changed files)** | 1 GET + ~7 PUTs | ref check + delta + ref update |
| **pull (fast-forward)** | ~3 GETs | ref + new objects |
| **pull (merge, no conflict)** | ~5 GETs | ref + objects + merge (local) |

| **read one file (cloneless)** | 2-4 GETs | ref → commit → tree(s) → blob |
| **status** | **0** | local only (content_hash comparison) |
| **check for pending** | 1 GET | read pending_counts.json |
