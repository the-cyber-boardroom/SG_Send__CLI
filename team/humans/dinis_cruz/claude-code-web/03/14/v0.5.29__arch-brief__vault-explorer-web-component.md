# Architect Brief: Vault Explorer Web Component

**Version:** v0.5.29
**Date:** 2026-03-14
**Type:** Architecture Brief
**Team:** Explorer → Architect

---

## Objective

Design a web component/tool that provides detailed, interactive visualisation of a vault's internal structure. The component should let users browse, inspect, and understand everything inside an encrypted vault — from the high-level tree down to individual encrypted blobs.

## Problem

Today, vault internals are only visible through CLI commands and raw JSON. There is no visual way to:

- Navigate the tree structure of a vault
- Inspect what a commit contains (tree reference, parent, signature, message)
- Click on an encrypted file and see its decrypted contents
- Understand the relationship between branches, commits, trees, and blobs
- Trace the chain from a branch head → commit → tree → blob → decrypted file

This makes debugging, onboarding, and trust-building harder than it needs to be.

## Scope

### Core Views

1. **Tree Explorer** — Interactive file/folder tree view
   - Renders `Schema__Object_Tree` as a navigable hierarchy
   - Each `Schema__Object_Tree_Entry` shows: `path`, `name`, `size`, `content_hash`
   - Click a `blob_id` → opens Blob Viewer
   - Click a `tree_id` → expands nested tree
   - Visual indicators for encrypted fields (`name_enc`, `size_enc`, `content_hash_enc`)

2. **Commit Inspector** — Detailed commit view
   - Renders `Schema__Object_Commit` fields: `parent(s)`, `tree_id`, `timestamp`, `message`, `version`, `branch_id`
   - PKI fields: `author_key_id`, `author_signature`, `signature`, `attestations`
   - Click `tree_id` → opens Tree Explorer for that commit's snapshot
   - Click `parent` → navigates to parent commit
   - Signature verification status (valid / invalid / unsigned)

3. **Blob Viewer** — Encrypted file content viewer
   - Shows raw ciphertext (hex or base64)
   - If vault key is available: decrypt and show plaintext
   - Metadata: `object_id`, size, encryption parameters (IV, auth tag)
   - Content-type detection for decrypted content (text, JSON, image, binary)

4. **Branch Navigator** — Branch overview
   - Renders `Schema__Branch_Index` and `Schema__Branch_Meta`
   - Shows: `branch_id`, `name`, `branch_type`, `head_ref_id`, `created_at`, `creator_branch`
   - PKI info: `public_key_id`, `private_key_id`
   - Click `head_ref_id` → opens Commit Inspector at branch head
   - Visual branch graph (like `git log --graph`)

5. **Vault Overview** — Landing page
   - Renders `Schema__Vault_Meta`: `vault_id`, `name`, `version`
   - Sync state from `Schema__Vault_Index`: file count, sync status per entry (`Enum__Sync_State`)
   - Links to Branch Navigator, Tree Explorer (at HEAD), Commit Inspector (latest)

### Navigation Model

```
Vault Overview
  ├── Branch Navigator
  │     └── click branch → Commit Inspector (at head)
  │           └── click tree_id → Tree Explorer
  │                 └── click blob_id → Blob Viewer
  │           └── click parent → Commit Inspector (parent)
  └── Tree Explorer (current HEAD)
        └── click blob_id → Blob Viewer
        └── click tree_id → Tree Explorer (nested)
```

### Data Source

The component reads vault data via the **SG/Send Transfer API** (or a local vault directory):

- **Remote mode:** Uses the Transfer API (`api/transfers/`) and Vault API (`api/vault/`) to fetch objects
- **Local mode:** Reads from `.sg_vault/bare/objects/` directory structure
- **Decryption:** Requires the vault key (`Safe_Str__Vault_Key`) to decrypt blobs and trees
  - Key is provided by the user (never stored by the component)
  - Decryption happens client-side using Web Crypto API (matching `Vault__Crypto` interop)

### Key Schemas the Component Must Understand

| Schema | Used In | Purpose |
|--------|---------|---------|
| `Schema__Vault_Meta` | Vault Overview | Identity and version |
| `Schema__Vault_Index` | Vault Overview | Sync state of all files |
| `Schema__Vault_Index_Entry` | Vault Overview | Per-file sync status |
| `Schema__Object_Tree` | Tree Explorer | Directory listing |
| `Schema__Object_Tree_Entry` | Tree Explorer | File/subtree metadata |
| `Schema__Object_Commit` | Commit Inspector | Snapshot with parent chain |
| `Schema__Branch_Index` | Branch Navigator | All branches |
| `Schema__Branch_Meta` | Branch Navigator | Per-branch identity + PKI |
| `Schema__PKI_Key_Pair` | Commit Inspector | Signature verification |

### Crypto Interop Requirements

The web component MUST use the same crypto parameters as `Vault__Crypto`:

- **AES-256-GCM** — 12-byte IV, 16-byte auth tag
- **PBKDF2** — 600,000 iterations, SHA-256, 32-byte key, salt = `sg-vault-v1:{vault_id}`
- **HKDF-SHA256** — Per-file key derivation from vault key
- **Object IDs** — SHA-256 of ciphertext, truncated to 12-char hex
- **Content Hash** — SHA-256 of plaintext, truncated to 12-char hex

These are already defined for Web Crypto API interop — the component should reuse the browser's native `crypto.subtle` API.

## Acceptance Criteria

1. **Tree browsing** — Can click through a tree hierarchy and see all entries with their metadata
2. **Commit inspection** — Can view any commit's full details and navigate parent chain
3. **Blob decryption** — Given a vault key, can decrypt and display file contents in-browser
4. **Branch navigation** — Can see all branches and jump to any branch's head commit
5. **Zero server-side decryption** — All decryption happens in the browser via Web Crypto API
6. **Schema compatibility** — Component correctly parses all vault schemas listed above
7. **Signature verification** — Shows whether commits are signed and if signatures are valid
8. **Breadcrumb navigation** — User always knows where they are: `vault > branch > commit > tree > blob`

## Non-Goals (for v1)

- Editing or writing vault contents (read-only viewer)
- Merge conflict resolution UI
- Multi-vault comparison
- Real-time sync / live updates
- Vault creation or key generation

## Open Questions for Architect

1. **Web component framework** — Vanilla web components, Lit, or something else? Must work embedded in SG/Send's existing web UI.
2. **State management** — How to handle navigation state (URL hash? internal router?) for deep-linking to specific commits/blobs.
3. **Lazy loading** — Large vaults may have thousands of objects. Should trees be fetched on-demand or pre-loaded?
4. **Key UX** — How should the vault key be provided? Paste-in field? URL fragment (never sent to server)? Session-only memory?
5. **Diff view** — Should v1 include a commit-to-commit diff view (comparing two trees), or defer to v2?
