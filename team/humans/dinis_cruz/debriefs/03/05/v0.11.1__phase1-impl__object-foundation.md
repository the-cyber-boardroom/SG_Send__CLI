## Key Implementation Details

### Object Store Layout
```
.sg_vault/
    objects/
        a1/b2c3d4e5f6    ← cached object (2-char prefix dirs, like git)
        b2/c3d4e5f6a1    ← another object
    refs/
        head              ← current commit_id (12 hex chars)
```

### Content Addressing
- `compute_object_id(ciphertext)` = `sha256(ciphertext).hexdigest()[:12]`
- Same plaintext encrypted twice → different ciphertext → different object_id (security > dedup)
- Object IDs match existing `Safe_Str__File_Id` server validation (12 hex chars)

### Legacy Vault Protection
- Detection: `.sg_vault/tree.json` exists AND `.sg_vault/refs/head` does NOT exist → legacy
- Raises `Legacy_Vault_Error` with clear message pointing to web interface
- Guards on: pull, push, status, remote_status
- Safe: clone (creates new format)

### Inspector CLI Commands
```bash
sg-send-cli inspect [directory]           # vault overview
sg-send-cli inspect-object <id> [dir]     # object detail + integrity
sg-send-cli inspect-tree [directory]      # tree entries
sg-send-cli inspect-log [directory]       # commit chain
sg-send-cli inspect-stats [directory]     # object store statistics
```

### Ref File ID Derivation
- Domain: `sg-vault-v1:file-id:ref:{vault_id}`
- Distinct from tree (`sg-vault-v1:file-id:tree:{vault_id}`) and settings domains
- Now included in `derive_keys()` return dict

## What's Next: Phase 2

Phase 2 will rewrite the core sync logic to use the object store:
- `push`: create blob/tree/commit objects → upload → update HEAD ref
- `pull`: download HEAD → walk commits → download missing objects
- `clone`: download HEAD → commit → tree → all blobs
- Conflict detection via parent-commit lineage
- SHA-256 content hashing for change detection (replace size-only)
